# 004 — Context: Cancellation and Timeout

## Контекст

В предыдущих задачах воркеры останавливались только одним способом — через закрытие канала `jobs`. Это graceful сигнал: "новых задач не будет, дочитай что есть и завершись". Но в продакшне этого недостаточно.

Представь: пользователь отменил HTTP запрос, или истёк дедлайн, или оператор хочет немедленно остановить сервис. В этих случаях ждать пока воркеры дочитают очередь нельзя — нужна **немедленная остановка**. Именно для этого в Go существует `context.Context`.

Контекст — это стандартный механизм распространения сигнала отмены через всю цепочку вызовов: от HTTP handler через сервисный слой до воркеров пула и SQL запросов в базе. Любая долгоживущая операция в продакшн коде обязана принимать контекст первым параметром.

В этой задаче ты впервые добавишь `select` с двумя case в воркер — и поймёшь почему без него воркер глух к любым внешним сигналам.

---

## Задача

Напиши программу в которой:

1. Функция `startWorker(ctx context.Context, id int, jobs <-chan int, results chan<- Result, wg *sync.WaitGroup)` — читает задачи из `jobs` через `select` с двумя case: один для задачи из канала, другой для `ctx.Done()`. Каждую задачу обрабатывает с паузой `50 * time.Millisecond` — имитация реальной работы. При завершении по любой причине печатает `"worker-<id> stopped: <причина>"`.

2. Тип `Result` — структура с полями `Value int` и `WorkerID int`.

3. В `main`:
    - Создай контекст с таймаутом `200 * time.Millisecond`.
    - Не забудь `defer cancel()`.
    - Создай канал `jobs` буферизованный на 20 элементов и отправь туда числа от 1 до 20.
    - Запусти 3 воркера передав им контекст.
    - Запусти горутину-надсмотрщика.
    - Собери результаты из `results` через `range`.
    - В конце напечатай сколько задач было обработано из 20.

---

## Требования

- Воркер использует `select` — не `if` и не отдельную проверку контекста в начале цикла.
- При остановке по контексту воркер печатает причину через `ctx.Err()`.
- При остановке по закрытию канала (`ok == false`) воркер печатает `"jobs closed"`.
- `defer cancel()` обязателен в `main` — даже если таймаут сработает раньше.
- Программа завершается чисто — без дедлока, без зависания.
- Никаких `time.Sleep` для синхронизации — только контекст и каналы.

---

## Ожидаемый вывод

Точный вывод зависит от планировщика, но структура всегда такая: несколько задач успевают обработаться за 200мс, потом контекст истекает и воркеры останавливаются. При паузе 50мс каждая задача и 3 воркерах — успеет обработаться примерно 9-12 задач.

```
worker-0 processed: 1 -> 1
worker-1 processed: 2 -> 4
worker-2 processed: 3 -> 9
worker-0 processed: 4 -> 16
...
worker-0 stopped: context deadline exceeded
worker-1 stopped: context deadline exceeded
worker-2 stopped: context deadline exceeded
processed 9 out of 20 tasks
main done
```

## Вопросы ДО написания кода

**Вопрос 1.**  
Рассмотри воркер без `select` — с простым `range`:

```go
for job := range jobs {
    time.Sleep(50 * time.Millisecond)
    results <- Result{Value: job * job}
}
```

Контекст отменяется через 200мс. Что произойдёт с этим воркером? Остановится ли он? Если нет — когда он всё-таки завершится и почему это плохо в продакшне?

**Вопрос 2.**  
`select` выбирает case случайно если несколько готовы одновременно. В воркере два case: `jobs` и `ctx.Done()`. В момент когда таймаут истекает, в `jobs` может ещё быть непрочитанные задачи. Что выберет `select`? Гарантированно ли воркер немедленно остановится или может взять ещё одну задачу перед выходом? Это баг или нормальное поведение?

**Вопрос 3.**  
`context.WithTimeout` и `context.WithDeadline` — в чём разница между ними семантически? Покажи как переписать `WithTimeout(parent, 200ms)` через `WithDeadline` с эквивалентным результатом.

**Вопрос 4.**  
Зачем нужен `defer cancel()` если таймаут всё равно истечёт сам? Что происходит с ресурсами контекста если `cancel` никогда не вызывается? Это утечка памяти? Насколько это критично?

**Вопрос 5.**  
Посмотри на этот select в воркере:

```go
select {
case job, ok := <-jobs:
    if !ok {
        return
    }
    // обработка
case <-ctx.Done():
    return
}
```

Что произойдёт если `jobs` закрыт И контекст отменён одновременно? Какой case выполнится? Имеет ли значение для корректности программы какой из них сработает?

**Вопрос 6.**  
`ctx.Err()` возвращает разные значения в зависимости от причины отмены. Какие значения возможны и при каком условии возникает каждое? Напиши в голове `switch` по `ctx.Err()` который обрабатывает каждый случай по-разному.

**Вопрос 7.**  
Горутина-надсмотрщик делает `wg.Wait()` а потом `close(results)`. Но контекст уже отменён — воркеры завершились быстро. Надсмотрщик дождётся их через WaitGroup и закроет `results`. `main` читает `range results` и завершится.

А теперь вопрос: что будет если воркер при остановке по контексту попытается отправить последний результат в `results`? Отправка в `results` тоже должна учитывать контекст — почему и как это реализовать?

## Эксперимент: два способа остановки

Это обязательная часть задачи.

Реализуй `main` дважды — в двух отдельных функциях — и запусти оба сценария последовательно.

**Сценарий А — остановка по таймауту:** таймаут 200мс, 20 задач по 50мс каждая, 3 воркера. Воркеры должны остановиться раньше чем обработают все задачи.

**Сценарий Б — graceful завершение:** таймаут 10 секунд (заведомо большой), 9 задач по 50мс каждая, 3 воркера. Все задачи должны обработаться, воркеры останавливаются через закрытие `jobs`, таймаут не срабатывает.

Запусти оба сценария и убедись что воркеры печатают разные причины остановки: в сценарии А — `"context deadline exceeded"`, в сценарии Б — `"jobs closed"`.

Это важно: один и тот же код воркера корректно обрабатывает оба пути завершения.

## Эксперименты после решения

**Эксперимент А.** Убери `defer cancel()` из `main`. Запусти с race detector: `go run -race .`. Есть ли предупреждения? Добавь в `main` `runtime.GC()` после основного кода — утечка контекста влияет на GC?

**Эксперимент Б.** Замени `context.WithTimeout` на `context.WithCancel`. Вызови `cancel()` вручную через `time.AfterFunc(200*time.Millisecond, cancel)`. Поведение идентично? В чём разница между этими подходами в реальном коде?

**Эксперимент В.** Добавь в воркер второй select при отправке результата:

```go
select {
case results <- Result{Value: job * job, WorkerID: id}:
case <-ctx.Done():
    return
}
```

Без этого второго select что происходит если `results` заполнен и контекст отменяется? Воркер зависнет? Проверь.

## Вопросы ПОСЛЕ решения

**Вопрос А.**  
Контекст образует дерево: родительский контекст может иметь дочерние. Если отменить родительский — отменятся ли дочерние? А если отменить дочерний — отменится ли родительский? Нарисуй дерево контекстов для типичного HTTP запроса: `Background -> Request -> DB query -> Worker`.

**Вопрос Б.**  
В твоём воркере `select` проверяет `jobs` и `ctx.Done()` **недетерминированно**. Представь что тебе нужен **приоритетный** выход по контексту — воркер должен немедленно выйти при отмене контекста, даже если в `jobs` есть задачи. Как реализовать это через двойной `select`? Напиши псевдокод.

**Вопрос В.**  
Стандартная библиотека Go передаёт контекст **первым параметром** по соглашению. Это не требование компилятора — это конвенция. Почему именно первым? Что будет если ты создашь пакет где контекст передаётся последним — это скомпилируется? Какие проблемы это создаст?

---

## Критерии правильного решения

Твоё решение корректно если: воркер использует `select` с двумя case — для задачи и для контекста; при таймауте воркеры останавливаются до обработки всех 20 задач; при graceful завершении воркеры обрабатывают все задачи и останавливаются через `ok == false`; отправка результата в `results` тоже защищена `select` с `ctx.Done()`; `defer cancel()` присутствует в `main`; программа завершается чисто в обоих сценариях.
