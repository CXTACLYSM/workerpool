# 010 — Финальный пул: Дженерики и всё вместе

## Контекст

Ты прошёл путь от одного воркера читающего строки из канала до структуры с метриками, graceful shutdown, backpressure и защитой от паник. Каждая задача добавляла один слой. Теперь пришло время собрать всё в единую обобщённую конструкцию.

До этого момента пул работал только с конкретным типом `Job`. В продакшне это ограничение неприемлемо: один сервис обрабатывает HTTP запросы, другой — записи из Kafka, третий — файлы для конвертации. Писать отдельный пул для каждого типа — это дублирование кода, разные баги в каждой копии, и бесконечный рефакторинг.

Дженерики решают эту проблему на уровне системы типов. Пул параметризуется типом результата `T` — и компилятор проверяет типы статически, без единого `interface{}` и без паник от type assertion в рантайме. Один код, бесконечная типобезопасность.

Это финальная задача серии. Она не добавляет новых концептов — она требует синтеза всего что ты понял за предыдущие девять шагов. Если что-то из прошлых задач осталось неясным — это проявится здесь. Это и есть цель.

## Задача

Напиши **полный, самодостаточный** пакет `workerpool` в файле `pool.go`. Рядом напиши `main.go` с двумя конкретными применениями которые используют один и тот же код пула.

### Типы и структуры

Определи обобщённый тип задачи: `Task[T any]` — это функция принимающая `context.Context` и возвращающая `(T, error)`. Никаких структур-обёрток — задача это просто функция.

Определи обобщённый результат: `Result[T any]` — структура с полями `Value T`, `Err error`, `WorkerID int`, `Duration time.Duration`. Поле `Duration` — сколько времени заняло выполнение задачи включая возможный recover.

Определи снимок метрик: `MetricsSnapshot` — структура с обычными `int64` полями `Processed`, `Errors`, `Panics`, `InFlight`, `QueueDepth`. Не использует `atomic` — это именно снимок для чтения.

Определи конфигурацию: `Config[T any]` — структура с полями `Workers int`, `QueueSize int`, `PanicHandler func(workerID int, recovered any, job Task[T])`.

### Структура Pool[T any]

Поля: `cfg Config[T]`, `jobs chan Task[T]`, `results chan Result[T]`, `wg sync.WaitGroup`, `ctx context.Context`, `cancel context.CancelFunc`, `once sync.Once`, вложенная структура с атомарными метриками.

### Методы Pool[T any]

`NewPool[T any](ctx context.Context, cfg Config[T]) *Pool[T]` — конструктор. Валидирует конфигурацию: если `Workers <= 0` или `QueueSize <= 0` — паникует с понятным сообщением. Создаёт дочерний контекст, запускает воркеров и надсмотрщика.

`Submit(task Task[T]) error` — блокирующая отправка. Возвращает `ErrPoolStopped` если пул завершён.

`TrySubmit(task Task[T]) bool` — неблокирующая попытка. Возвращает `false` если очередь полна или пул остановлен.

`SubmitWithTimeout(task Task[T], timeout time.Duration) error` — отправка с таймаутом. Использует `time.NewTimer` с явным `Stop()`.

`Results() <-chan Result[T]` — read-only канал результатов.

`Metrics() MetricsSnapshot` — снимок метрик в момент вызова.

`QueueDepth() int` — текущая длина очереди.

`Shutdown()` — graceful завершение. Защищён `sync.Once`. Закрывает `jobs`, ждёт воркеров.

`Stop()` — немедленная остановка через `cancel()`. Ждёт воркеров.

Приватный метод `executeTask(workerID int, task Task[T]) Result[T]` — выполняет задачу с recover, замером времени и обновлением метрик.

## Два применения в main.go

### Применение А — параллельный fetch URL

Имитируй HTTP запросы к URLs. Каждый "запрос" — это `time.Sleep` случайной длительности от 10 до 100ms плюс случайная ошибка с вероятностью 20%. Один URL из списка всегда паникует — имитация бага в парсере ответа.

```go
type FetchResult struct {
    URL        string
    StatusCode int
    Body       string
}

urls := []string{
    "https://api.example.com/users",
    "https://api.example.com/products",
    "https://api.example.com/orders",
    // ... 10 URLs
    "https://api.example.com/panic", // этот всегда паникует
}

pool := workerpool.NewPool[FetchResult](ctx, workerpool.Config[FetchResult]{
    Workers:   4,
    QueueSize: len(urls),
    PanicHandler: func(id int, r any, task workerpool.Task[FetchResult]) {
        log.Printf("PANIC in worker %d: %v", id, r)
    },
})
```

Собери результаты. Напечатай успешные fetches, ошибки и пойманные паники отдельно.

### Применение Б — параллельная обработка чисел

Тот же код пула, другой тип результата. Обработай список из 20 чисел: для каждого вычисли является ли оно простым (используй наивный алгоритм с паузой имитирующей тяжёлые вычисления). Числа 0 и 1 возвращают ошибку — не являются простыми по определению.

```go
numbers := []int{0, 1, 2, 3, 4, 17, 100, 97, ...}

pool := workerpool.NewPool[bool](ctx, workerpool.Config[bool]{
    Workers:   3,
    QueueSize: len(numbers),
})
```

Напечатай для каждого числа: простое оно или нет, и сколько времени заняло вычисление.

## Требования

Весь код пула живёт в одном файле `pool.go` без внешних зависимостей — только стандартная библиотека. Дженерики используются корректно: нигде нет `interface{}`, нигде нет type assertion. Оба применения используют один и тот же бинарный код пула без модификаций. Детектор гонок молчит на обоих применениях. После завершения всех задач `InFlight == 0` в обоих применениях. `PanicHandler` вызывается при панике в применении А.

## Вопросы ДО написания кода

**Вопрос 1.**
Дженерик функция `NewPool[T any]` инстанциируется компилятором отдельно для каждого конкретного типа `T`. Это называется **monomorphization**. Сколько версий `executeTask` будет в скомпилированном бинарнике если ты используешь пул с `FetchResult` и `bool`? Чем это отличается от подхода Java с type erasure где дженерики существуют только на уровне компилятора и стираются в рантайме?

**Вопрос 2.**
`Task[T any]` определён как тип функции: `type Task[T any] func(ctx context.Context) (T, error)`. Почему задача принимает `context.Context` как параметр, а не захватывает его из замыкания? Рассмотри конкретный сценарий: пул создаётся с контекстом с таймаутом 5 секунд, но конкретная задача должна завершиться за 500ms. Как передача контекста через параметр позволяет реализовать это элегантно?

**Вопрос 3.**
`PanicHandler` принимает `task Task[T]` — саму запаниковавшую задачу. Зачем? Ведь при панике мы уже знаем что что-то пошло не так. Придумай конкретный продакшн-сценарий где доступ к оригинальной задаче в обработчике паники критически важен.

**Вопрос 4.**
Конструктор валидирует конфигурацию и паникует при невалидных значениях: `if cfg.Workers <= 0 { panic("workers must be > 0") }`. Это намеренное использование `panic` — не `error`. Когда в Go уместно возвращать `error` из конструктора, а когда — паниковать? Какой принцип определяет этот выбор?

**Вопрос 5.**
В `executeTask` ты замеряешь `Duration` — время выполнения задачи. Это время включает возможный `recover` от паники. Если задача паникует немедленно — `Duration` будет очень маленьким. Если задача выполняется 5 секунд а потом паникует — `Duration` будет ~5 секунд. Для чего эта метрика полезна в продакшне? Приведи конкретный пример алертинга основанного на `Duration`.

**Вопрос 6.**
Ты реализуешь два применения — `Pool[FetchResult]` и `Pool[bool]`. Компилятор генерирует отдельный код для каждого. Но `sync.WaitGroup`, `sync.Once`, `atomic.Int64` внутри пула — они не параметризованы типом `T`. Как компилятор обрабатывает эти поля при monomorphization? Дублируется ли их код тоже или они разделяются между инстанциями?

## Критерии правильного решения

Это финальная задача — критерии строже. Твоё решение полностью корректно если выполняются все условия одновременно.

По коду: нет ни одного `interface{}` в публичном API; нет type assertion нигде в `pool.go`; `executeTask` содержит recover, замер времени и обновление всех четырёх метрик; `Shutdown` защищён `sync.Once`; `SubmitWithTimeout` использует `time.NewTimer` с `Stop()`; конструктор валидирует конфигурацию.

По поведению: оба применения работают с одним кодом пула без изменений; паника в применении А поймана и передана в `PanicHandler`; `InFlight == 0` после завершения в обоих применениях; стресс-тест проходит с `-race` без предупреждений.

По архитектуре: воркер не логирует ничего самостоятельно — только передаёт данные; задача принимает контекст параметром а не из замыкания; публичный API минимален — только то что нужно пользователю пула.
