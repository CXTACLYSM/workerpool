# 008 — Submit: Backpressure и SubmitWithTimeout

## Контекст

В задаче 007 ты добавил `Submit` который блокируется когда очередь полна. Это называется **backpressure** — давление назад. Когда система перегружена, она не принимает новые данные: она сигнализирует отправителю "притормози". Это не баг, это механизм саморегуляции.

Представь трубу с водой. Если кран открыт сильнее чем труба может пропустить — вода давит назад на источник. В программировании backpressure работает так же: отправитель блокируется когда получатель не успевает обрабатывать. Это лучше чем альтернативы: безграничный буфер (переполнение памяти), тихое отбрасывание задач (потеря данных), или паника (смерть сервиса).

Но иногда блокировать отправителя бесконечно тоже нельзя. HTTP handler не может ждать вечно — у него есть дедлайн клиента. Batch процессор хочет знать что система перегружена и залогировать это. Для таких случаев нужен `SubmitWithTimeout`: попробуй поставить задачу в очередь, но если не получилось за отведённое время — сообщи об этом явно.

В этой задаче ты реализуешь оба метода с тонкой семантикой каждого, и впервые увидишь как backpressure распространяется через систему снизу вверх — от воркеров к очереди к отправителю.

## Задача

Расширь структуру `Pool` из задачи 007. Добавь или измени следующие методы:

1. Метод `Submit(job Job) error` — блокирующая отправка. Ждёт места в очереди сколько угодно. Возвращает ошибку только если пул остановлен. Никогда не отбрасывает задачу молча.

2. Метод `TrySubmit(job Job) bool` — неблокирующая попытка. Если в очереди есть место прямо сейчас — кладёт задачу и возвращает `true`. Если очередь полна — возвращает `false` немедленно, не блокируясь ни на миллисекунду. Не возвращает ошибку — только успех или неуспех.

3. Метод `SubmitWithTimeout(job Job, timeout time.Duration) error` — отправка с таймаутом. Ждёт места в очереди не дольше `timeout`. Если место появилось — отправляет и возвращает `nil`. Если таймаут истёк раньше — возвращает конкретную ошибку с информацией о заполненности очереди. Если пул остановлен — возвращает ошибку немедленно.

4. В `main` — реализуй три сценария демонстрирующих каждый метод.

---

## Требования

- `Submit` не содержит `default` в `select` — это принципиально. Наличие `default` превратило бы его в неблокирующий и сломало бы семантику backpressure.
- `TrySubmit` содержит `default` — это единственный случай где `default` уместен.
- `SubmitWithTimeout` использует `time.NewTimer` а не `time.After` — объясни почему в вопросах до написания.
- Ошибка от `SubmitWithTimeout` при таймауте содержит текущую заполненность очереди: `"submit timeout after 100ms: queue %d/%d"`.
- Все три метода корректно обрабатывают остановленный пул.
- Детектор гонок `go run -race .` молчит.

## Сценарии в main

**Сценарий А — Backpressure через Submit:**

Создай пул с 2 воркерами и очередью размером 3. Каждая задача занимает `100ms`. Запусти горутину которая отправляет 10 задач через `Submit` и замеряет время каждой отправки. Ты увидишь что первые 3 отправки мгновенные (заполняют буфер), потом каждая следующая блокируется на ~100ms (ждёт воркера). Напечатай время каждого `Submit`.

```
[Scenario A: Backpressure]
submit job-0: 0ms   (буфер свободен)
submit job-1: 0ms   (буфер свободен)
submit job-2: 0ms   (буфер свободен)
submit job-3: 98ms  (ждал воркера)
submit job-4: 101ms (ждал воркера)
...
```

**Сценарий Б — TrySubmit при перегрузке:**

Тот же пул. Попробуй отправить 10 задач через `TrySubmit` не дожидаясь освобождения очереди. Посчитай сколько задач удалось поставить сразу, сколько было отклонено. Напечатай статистику.

```
[Scenario B: TrySubmit]
accepted: 3 / 10
rejected: 7 / 10
```

**Сценарий В — SubmitWithTimeout:**

Новый пул с 1 воркером и очередью размером 2. Задача занимает `200ms`. Отправь 5 задач с таймаутом `150ms` на каждую. Первые две попадут в буфер мгновенно, третья заблокируется и истечёт по таймауту — напечатай ошибку с заполненностью. Остальные тоже.

```
[Scenario C: SubmitWithTimeout]
job-0: submitted OK
job-1: submitted OK
job-2: error: submit timeout after 150ms: queue 2/2
job-3: error: submit timeout after 150ms: queue 2/2
job-4: error: submit timeout after 150ms: queue 2/2
```

## Вопросы ДО написания кода

**Вопрос 1.**
Рассмотри два варианта `Submit` и объясни принципиальную разницу в поведении:

```go
// Вариант А — с default
func (p *Pool) Submit(job Job) error {
    select {
    case p.jobs <- job:
        return nil
    case <-p.ctx.Done():
        return ErrPoolStopped
    default:
        return ErrQueueFull // тихо отбрасывает задачу
    }
}

// Вариант Б — без default
func (p *Pool) Submit(job Job) error {
    select {
    case p.jobs <- job:
        return nil
    case <-p.ctx.Done():
        return ErrPoolStopped
    }
}
```

В каком сценарии вариант А кажется привлекательным? Почему он всё равно неправильный для метода называющегося `Submit`? Какой метод из твоего задания соответствует варианту А семантически?

**Вопрос 2.**
`time.After(duration)` и `time.NewTimer(duration)` — оба создают таймер. В чём разница и почему в `SubmitWithTimeout` нужен именно `NewTimer`? Что происходит с внутренним таймером `time.After` если `select` завершается раньше чем таймер истечёт — через case `p.jobs` или `ctx.Done()`?

**Вопрос 3.**
`TrySubmit` возвращает `bool` а не `error`. Это намеренное решение. Чем семантически отличается "очередь полна" от "пул остановлен"? Должен ли `TrySubmit` возвращать разные значения в этих двух случаях? Как ты обработаешь оба случая в реализации?

**Вопрос 4.**
В сценарии А ты замеряешь время каждого `Submit`. Предположим воркер обрабатывает задачу за 100ms, воркеров 2, буфер 3. Сколько времени займёт отправка задачи номер 4 (индекс с нуля)? Задачи 5? Задачи 9? Посчитай в голове перед запуском и сравни с реальным результатом.

**Вопрос 5.**
Backpressure блокирует отправителя. В HTTP сервере отправитель — это handler горутина. Если handler заблокирован в `Submit` на 500ms — что происходит с HTTP соединением клиента? С другими входящими запросами? С пулом горутин самого HTTP сервера? Опиши цепочку последствий.

**Вопрос 6.**
Посмотри на этот код `SubmitWithTimeout` и найди утечку ресурсов:

```go
func (p *Pool) SubmitWithTimeout(job Job, timeout time.Duration) error {
    select {
    case p.jobs <- job:
        return nil
    case <-time.After(timeout): // утечка?
        return fmt.Errorf("timeout")
    case <-p.ctx.Done():
        return ErrPoolStopped
    }
}
```

`time.After` создаёт таймер который будет тикать до истечения `timeout` даже если `select` уже завершился через другой case. Если `SubmitWithTimeout` вызывается 10 000 раз в секунду с таймаутом 30 секунд — сколько живых таймеров будет в памяти одновременно? Как `time.NewTimer` с явным `Stop()` решает эту проблему?

## Вопросы ПОСЛЕ решения

**Вопрос А.**
Backpressure — это паттерн из реактивного программирования. В Go он реализуется через блокирующие каналы. В других системах backpressure реализован иначе: TCP использует окно перегрузки, Kafka имеет `max.block.ms` у producer, gRPC имеет flow control. Что общего у всех этих механизмов на концептуальном уровне?

**Вопрос Б.**
`TrySubmit` возвращает `false` при полной очереди. Вызывающий код получает `false` и должен решить что делать: повторить попытку, залогировать, вернуть ошибку клиенту. Это называется **explicit rejection**. Чем этот подход отличается от **implicit dropping** (когда канал с `default` тихо выбрасывает задачу)? В каких продакшн-системах explicit rejection предпочтительнее?

**Вопрос В.**
`time.NewTimer` требует явного `Stop()` для предотвращения утечки. Но `Stop()` возвращает `bool` — `true` если таймер был остановлен до срабатывания, `false` если уже сработал. В некоторых случаях нужно дренировать канал таймера после `Stop()`. Напиши правильный паттерн использования `NewTimer` с корректным `Stop()` и дренированием канала.

## Критерии правильного решения

Твоё решение корректно если выполняются все условия одновременно: `Submit` не имеет `default` и реально блокируется при полной очереди; `TrySubmit` возвращает результат мгновенно — никогда не блокируется; `SubmitWithTimeout` использует `time.NewTimer` с явным `Stop()` при успехе; ошибка таймаута содержит текущую заполненность очереди; в сценарии А видна задержка начиная с четвёртой задачи; детектор гонок молчит на всех трёх сценариях.
