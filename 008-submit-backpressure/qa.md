# QA — 008 Submit: Backpressure и три метода отправки

Этот файл строится поверх QA-001 — QA-007. Здесь только новое что появилось в задаче 008: три семантически разных метода отправки, backpressure как архитектурный паттерн, утечка таймеров и горячий путь. Отвечай вслух до чтения ответа.

---

## Блок 1: Backpressure — давление назад

**Q: Что такое backpressure и почему блокировка отправителя это особенность, а не баг?**

A: Backpressure — это механизм саморегуляции системы под нагрузкой. Когда получатель не успевает обрабатывать данные, он сигнализирует отправителю "притормози" — через блокировку. Отправитель не может продолжить пока система не освободит место. Это не баг потому что альтернативы хуже: безграничный буфер приводит к переполнению памяти, тихое отбрасывание задач — к потере данных, паника — к смерти сервиса. Блокировка отправителя — это честный сигнал "система перегружена". В Go небуферизованный канал реализует максимальный backpressure (каждая отправка ждёт получателя), буферизованный — ограниченный (ждёт только когда буфер полон). Именно поэтому `Submit` без `default` в `select` — это не забытый код, это намеренный архитектурный выбор.

---

**Q: Буфер канала освобождается когда воркер взял задачу или когда завершил её обработку?**

A: При взятии из канала — в момент когда воркер читает значение через `case job := <-p.jobs`. С этого момента значение больше не занимает место в буфере и канал готов принять следующее. Именно поэтому в сценарии А первые несколько задач проходят мгновенно: воркеры берут задачи из буфера быстро, освобождая место для следующих. Блокировка наступает только когда оба воркера заняты обработкой И буфер полон одновременно. Это важное понимание — "ёмкость системы" это `workers + jobsBuf`, а не просто `jobsBuf`.

---

**Q: В сценарии А с 2 воркерами, буфером 3 и задачей 100ms — первые 5 отправок оказались мгновенными, а не 3. Почему?**

A: Потому что воркеры берут задачи из буфера быстрее чем горутина-отправитель успевает заполнить его. Горутина-отправитель и воркеры работают конкурентно. Пока отправитель кладёт задачи 1-3 в буфер, воркеры уже берут job-1 и job-2 освобождая два слота. К моменту когда отправитель доходит до job-4 и job-5 — в буфере снова есть место. Теоретический пример в README описывает идеализированный сценарий где отправитель работает быстрее воркеров. В реальности поведение зависит от скорости планировщика и конкретного железа. Это нормально — важно понимать механизм, а не совпасть с конкретными числами.

---

**Q: Как `Submit` реализует backpressure на уровне кода — покажи связь между отсутствием `default` и блокировкой отправителя.**

A: Второй `select` в `Submit` не имеет `default`:

```go
select {
case p.jobs <- job:   // отправляем если есть место
    return nil
case <-p.ctx.Done():  // или выходим если пул остановлен
    return p.ctx.Err()
}
```

Без `default` `select` блокируется пока хотя бы один `case` не станет готовым. Если буфер `p.jobs` полон — `case p.jobs <- job` не готов. Если контекст активен — `case <-p.ctx.Done()` тоже не готов. Горотина-отправитель встаёт и ждёт. Это и есть backpressure: отправитель физически не может продолжить пока система не освободит место. Если добавить `default: return ErrQueueFull` — это разрушило бы backpressure: отправитель немедленно получал бы отказ вместо ожидания. Это была бы другая семантика — explicit rejection, не backpressure.

## Блок 2: Три метода отправки — семантические различия

**Q: Назови три метода отправки задачи, их сигнатуры и одним предложением семантику каждого.**

A: `Submit(job Job) error` — блокирующая отправка: ждёт места в очереди сколько угодно, возвращает ошибку только если пул остановлен, гарантирует что задача будет поставлена. `TrySubmit(job Job) bool` — неблокирующая попытка: возвращает `true` если место нашлось прямо сейчас, `false` немедленно если очередь полна или пул остановлен, никогда не ждёт. `SubmitWithTimeout(job Job, timeout time.Duration) error` — отправка с дедлайном: ждёт не дольше `timeout`, возвращает `nil` если успела, ошибку с заполненностью очереди если время истекло.

---

**Q: Почему `TrySubmit` возвращает `bool` а не `error`? Какую информацию теряет вызывающий код и когда это приемлемо?**

A: `bool` упрощает API для случая когда вызывающему коду не важно почему отправка не удалась — важен только факт успеха или неудачи. Но теряется различие между двумя принципиально разными ситуациями: `false` может означать "очередь полна, попробуй позже" или "пул остановлен, не пробуй больше". Это приемлемо в сценариях где логика одинакова в обоих случаях — например batch-процессор который просто считает сколько задач удалось поставить. В продакшн коде где нужно различать эти случаи сигнатура была бы `TrySubmit(job Job) (bool, error)` где `error != nil` означает пул остановлен а `false, nil` означает очередь полна.

---

**Q: Рассмотри два варианта `Submit` и объясни принципиальную разницу:**

```go
// Вариант А — с default
func (p *Pool) Submit(job Job) error {
    select {
    case p.jobs <- job:
        return nil
    case <-p.ctx.Done():
        return ErrPoolStopped
    default:
        return ErrQueueFull
    }
}

// Вариант Б — без default
func (p *Pool) Submit(job Job) error {
    select {
    case p.jobs <- job:
        return nil
    case <-p.ctx.Done():
        return ErrPoolStopped
    }
}
```

A: Вариант А — это `TrySubmit` под именем `Submit`. Он никогда не блокируется: если очередь полна, немедленно возвращает ошибку. Backpressure полностью отсутствует — отправитель не притормаживается, он просто получает отказ и должен сам решить что делать (повторить? выбросить задачу?). Это explicit rejection. Вариант Б — настоящий `Submit` с backpressure: отправитель блокируется и ждёт. Система саморегулируется. Называть метод `Submit` и реализовывать его как вариант А — это нарушение принципа наименьшего удивления: пользователь ожидает блокирующей семантики по имени, а получает неблокирующую. Это скрытый баг который проявится под нагрузкой когда задачи начнут молча теряться.

---

**Q: `SubmitWithTimeout` — в каком продакшн-сценарии он предпочтительнее `Submit`?**

A: HTTP handler не может ждать бесконечно — у него есть дедлайн клиента. Если `Submit` заблокируется на 30 секунд пока очередь не освободится, HTTP соединение давно закрылось и клиент получил таймаут. С `SubmitWithTimeout` handler может передать клиентский таймаут как параметр: "если не удалось поставить задачу за время жизни запроса — верни 503 Service Unavailable". Это позволяет системе деградировать предсказуемо под нагрузкой вместо того чтобы накапливать заблокированные handler-горутины которые в итоге исчерпают пул горутин HTTP сервера.

## Блок 3: time.NewTimer и утечка таймеров

**Q: Что такое "горячий путь" и почему на нём важно использовать `time.NewTimer` вместо `time.After` (до Go 1.23)?**

A: Горячий путь — это участок кода который выполняется очень часто, тысячи или десятки тысяч раз в секунду. В worker pool это тело воркера и методы отправки задач. `time.After(d)` до Go 1.23 создавал внутренний таймер который не мог быть собран GC до своего срабатывания. Если `SubmitWithTimeout` вызывается 10 000 раз в секунду с таймаутом 30 секунд, и большинство вызовов завершается раньше через `case p.jobs <- job` — в памяти накапливается до 300 000 живых таймеров ожидающих срабатывания. Это реальная утечка памяти. `time.NewTimer` с явным `Stop()` останавливает таймер немедленно при успешной отправке — ресурс освобождается сразу, не через 30 секунд.

---

**Q: Почему `defer timer.Stop()` правильно работает в обоих путях выхода из `SubmitWithTimeout`?**

A: При успешной отправке (`case p.jobs <- job`) — таймер ещё не сработал. `defer timer.Stop()` останавливает его немедленно при выходе из функции. Ресурс освобождается сразу — это именно то что мы хотим. При таймауте (`case <-timer.C`) — таймер уже сработал и его канал был прочитан. `timer.Stop()` на уже сработавшем таймере безопасен — он просто возвращает `false` (не успел остановить). В обоих случаях `defer` делает правильное — либо останавливает активный таймер, либо безвредно вызывает Stop на завершённом. Это именно тот паттерн который делает `defer` с `Stop()` идиоматичным.

---

**Q: Начиная с Go 1.23 `time.After` безопасен на горячем пути — GC умеет собирать незасработавшие таймеры. Есть ли всё ещё причины предпочитать `time.NewTimer` с явным `Stop()`?**

A: Да, две причины. Первая — явный контроль над моментом освобождения ресурса. GC соберёт таймер "когда-нибудь" — при очень высокой частоте вызовов (десятки тысяч в секунду) разница между "освобождено сразу" и "освобождено при следующем GC цикле" может быть заметна в метриках памяти и паузах GC. `timer.Stop()` даёт детерминированное немедленное освобождение. Вторая — совместимость: если код должен работать на Go < 1.23, `time.After` на горячем пути реальная проблема. Для новых проектов только на Go 1.23+ `time.After` вполне допустим — документация это явно говорит. Знать оба варианта и уметь объяснить разницу — это то что проверяют на собесах.

---

**Q: `Stop()` на `time.Timer` возвращает `bool`. Что означает `false` и нужно ли дренировать канал таймера?**

A: `Stop()` возвращает `false` если таймер уже сработал до вызова `Stop` — то есть значение уже было отправлено в канал `timer.C`. В этом случае канал содержит непрочитанное значение. Если после `Stop()` снова использовать тот же таймер через `Reset()` — в канале будет "старое" значение которое немедленно разблокирует следующий `select`. Правильный паттерн для повторного использования таймера:

```go
if !timer.Stop() {
    select {
    case <-timer.C: // дренируем канал
    default:        // если уже прочитан — не блокируемся
    }
}
timer.Reset(newDuration)
```

В `SubmitWithTimeout` таймер используется один раз и не переиспользуется — дренирование не нужно. Но для таймеров которые сбрасываются через `Reset` этот паттерн обязателен.

## Блок 4: Архитектура трёх методов и их взаимодействие

**Q: Все три метода содержат первый `select` с `default` для проверки контекста. Зачем эта проверка если второй `select` тоже проверяет `ctx.Done()`?**

A: Это оптимизация раннего выхода — "fast path". Если контекст уже отменён, нет смысла входить в блокирующий `select` и ждать планировщика. Первый `select` с `default` — неблокирующая проверка которая занимает наносекунды: "отменён ли контекст прямо сейчас?" Если да — немедленный возврат. Если нет — идём в основной `select`. Это не устраняет необходимость второго `case <-p.ctx.Done()` в основном `select` — между первой проверкой и попыткой отправки контекст может отмениться. Оба `select` нужны: первый — оптимизация, второй — корректность.

---

**Q: Может ли `Submit` получить панику если `Shutdown` уже закрыл `p.jobs`? Как это проявляется и как защититься?**

A: Да. Если `Shutdown` закрыл `p.jobs` но контекст ещё не отменён — первый `select` в `Submit` пройдёт через `default` (контекст активен). Второй `select` попытается `p.jobs <- job` — отправка в закрытый канал паникует с `panic: send on closed channel`. В текущей реализации это предотвращается архитектурно: горутина-отправитель которая вызывает `Shutdown` через `defer` — единственный источник `Submit` вызовов, и после завершения цикла она больше не вызывает `Submit`. Но если `Submit` и `Shutdown` вызываются из разных мест — нужна явная защита через атомарный флаг `closed` или проверку `len(p.jobs) == 0 && isClosed`.

## Блок 5: Паттерны и продакшн применение

**Q: Backpressure блокирует горутину-отправителя. В HTTP сервере отправитель — это handler-горутина. Опиши цепочку последствий если `Submit` заблокируется на 500ms.**

A: Цепочка разворачивается так. Handler-горутина блокируется в `Submit` на 500ms — она не может вернуть ответ клиенту. HTTP соединение держится открытым. Клиент ждёт или получает таймаут. Go HTTP сервер запускает новую горутину для каждого входящего запроса — при высокой нагрузке накапливаются сотни заблокированных handler-горутин. Каждая горотина занимает минимум 2KB стека. При тысячах заблокированных горотин — мегабайты памяти потрачены на ожидание. GC давление растёт. Новые запросы продолжают поступать и создавать новые горотины. В итоге система исчерпывает память или становится неотзывчивой. Именно поэтому в HTTP контексте `SubmitWithTimeout` с коротким таймаутом предпочтительнее `Submit` — система деградирует предсказуемо (503 клиенту) а не накапливает заблокированные горутины.

---

**Q: `TrySubmit` — это паттерн explicit rejection. Чем он отличается от implicit dropping и когда каждый из них уместен?**

A: Explicit rejection — вызывающий код явно получает `false` и сам решает что делать: залогировать, повторить, вернуть ошибку клиенту, инкрементировать метрику. Задача не потеряна молча — потеря зафиксирована. Implicit dropping — задача тихо выбрасывается внутри системы без уведомления отправителя. Это происходит например при `select { case ch <- v: default: }` где `default` просто ничего не делает. Explicit rejection предпочтителен когда каждая задача важна и потеря должна быть заметна в метриках и логах. Implicit dropping уместен только в специфических сценариях: метрики (потеря одного sample не критична), буферы событий где устаревшие события бессмысленны, fire-and-forget уведомления. В worker pool для бизнес-задач всегда explicit rejection.

---

**Q: Сценарий Б показал `accepted: 3 / 10` при буфере 3 и 2 воркерах. Почему не `accepted: 5 / 10` (буфер + воркеры)?**

A: Потому что `TrySubmit` — мгновенная операция без ожидания. В момент вызовов `TrySubmit` в цикле воркеры ещё не успели взять задачи из буфера — цикл выполняется намного быстрее чем воркеры успевают реагировать. Все 10 вызовов `TrySubmit` происходят практически одновременно с точки зрения планировщика. Буфер вмещает 3 задачи — первые 3 попытки успешны, остальные 7 видят полный буфер и возвращают `false`. Воркеры начинают брать задачи позже — когда цикл уже завершился. Это демонстрирует важное свойство `TrySubmit`: он отражает состояние системы прямо сейчас, а не через несколько миллисекунд.
