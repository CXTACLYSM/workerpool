# 005 — Error Handling: Errors as Data

## Контекст

До этого момента задачи в воркерах всегда успешно завершались. В продакшне это не так: HTTP запрос может упасть с таймаутом, запись в базу может вернуть constraint violation, внешний сервис может быть недоступен.

Ключевая философия Go в отношении ошибок: **ошибка — это значение, а не исключение**. Это означает что ошибка это просто данные которые передаются через те же механизмы что и любые другие данные — в нашем случае через канал `results`. Воркер не паникует, не логирует самостоятельно и не принимает решений за вызывающий код. Он честно возвращает ошибку и продолжает работать.

Это фундаментальный сдвиг мышления по сравнению с исключениями в других языках. В Go нет `try/catch`. Есть возврат `error` как второго значения — и explicit проверка на каждом уровне.

В этой задаче ты впервые построишь воркер который разделяет успешные результаты и ошибки, передаёт и то и другое через один канал, и позволяет вызывающему коду самому решать что делать с каждым случаем.

---

## Задача

Напиши программу в которой:

1. Тип `Job` — структура с полем `ID int` и полем `Value int`.

2. Тип `Result` — структура с полями `Job Job`, `Value int`, `Err error` и `WorkerID int`. Поле `Err` заполняется при ошибке, `Value` — при успехе. Получатель обязан проверить `Err` перед использованием `Value`.

3. Функция `process(job Job) (int, error)` — возвращает ошибку если `job.ID` делится на 3 (то есть задачи с ID 3, 6, 9, 12, 15 возвращают ошибку). В остальных случаях возвращает `job.Value * job.Value`.

4. Функция `startWorker(ctx context.Context, id int, jobs <-chan Job, results chan<- Result, wg *sync.WaitGroup)` — читает задачи через `select`, вызывает `process`, отправляет `Result` в канал независимо от того успех это или ошибка. Воркер **не логирует ошибки** — он передаёт их дальше.

5. В `main`:
    - Отправь 15 задач с ID от 1 до 15, `Value` равен `ID`.
    - Запусти 3 воркера с контекстом (таймаут 5 секунд — заведомо большой).
    - Собери все результаты.
    - Раздели их на успешные и ошибочные.
    - Напечатай каждую группу отдельно.
    - Напечатай итоговую статистику: сколько успешных, сколько ошибок.

---

## Требования

- `process` возвращает `fmt.Errorf("job %d failed: value is divisible by 3", job.ID)` — конкретное сообщение, не просто `errors.New("error")`.
- Воркер не содержит ни одного вызова `log` или `fmt` для ошибок — только передача через канал.
- Ошибка в одной задаче не останавливает воркера — он берёт следующую задачу.
- `Result` всегда содержит исходный `Job` — чтобы получатель знал какая именно задача упала.
- Итоговое число успешных задач: 10. Ошибочных: 5 (ID: 3, 6, 9, 12, 15).

## Вопросы ДО написания кода

**Вопрос 1.**
В Go принято возвращать ошибку как последнее значение из функции: `return value, err`. Почему именно последнее, а не первое? Это требование компилятора или конвенция? Что произойдёт если поменять порядок — `return err, value`?

**Вопрос 2.**
Рассмотри два подхода к обработке ошибок в воркере и объясни почему второй правильнее:

```go
// Подход А: воркер логирует ошибку и продолжает
result, err := process(job)
if err != nil {
    log.Printf("worker-%d error: %v", id, err)
    continue
}
results <- Result{Value: result}

// Подход Б: воркер передаёт ошибку как данные
result, err := process(job)
results <- Result{Job: job, Value: result, Err: err, WorkerID: id}
```

Почему подход А нарушает принцип разделения ответственности? Кто должен решать что делать с ошибкой — воркер или вызывающий код?

**Вопрос 3.**
`fmt.Errorf("job %d failed: %w", job.ID, originalErr)` — что означает глагол `%w` в отличие от `%v`? В чём практическая разница для вызывающего кода? Как использовать `errors.Is` и `errors.As` с обёрнутыми ошибками?

**Вопрос 4.**
Посмотри на этот код получателя и найди проблему:

```go
for r := range results {
    if r.Err != nil {
        fmt.Println("error:", r.Err)
        continue
    }
    fmt.Println("success:", r.Value) // безопасно ли это?
}
```

Что если кто-то изменит `process` так что она возвращает ненулевой `Value` вместе с ненулевым `Err`? Как сделать тип `Result` более защищённым от такого использования?

**Вопрос 5.**
В этой задаче воркер отправляет результат в `results` одной строкой — независимо от того ошибка это или успех:

```go
results <- Result{Job: job, Value: result, Err: err, WorkerID: id}
```

Что произойдёт с этой строкой если контекст отменяется в момент когда воркер пытается отправить в заполненный `results`? Нужен ли здесь `select` с `ctx.Done()`? Добавь его в своё решение.

**Вопрос 6.**
Ты собираешь результаты в два слайса: `successful []Result` и `failed []Result`. Несколько горутин пишут в `results`, но читает только одна — `main`. Нужен ли мьютекс для защиты этих слайсов? Почему?

## Эксперименты после решения

**Эксперимент А.** Измени `process` так чтобы она возвращала ошибку случайно с вероятностью 30% — используй `rand.Intn(10) < 3`. Запусти программу 5 раз. Статистика меняется? Сумма `successful + errors` всегда равна 15?

**Эксперимент Б.** Добавь в `process` случайную задержку: `time.Sleep(time.Duration(rand.Intn(50)) * time.Millisecond)`. Изменится ли порядок результатов? Всегда ли задача с меньшим ID приходит раньше? Почему нет?

## Вопросы ПОСЛЕ решения

**Вопрос А.**
Интерфейс `error` в Go определён как:

```go
type error interface {
    Error() string
}
```

Это самый маленький интерфейс в стандартной библиотеке — один метод. Почему именно интерфейс, а не конкретный тип? Что это даёт для расширяемости системы ошибок?

**Вопрос Б.**
В продакшне часто нужно не просто знать что ошибка произошла, но и **какого типа** ошибка. Например, различать `NotFoundError`, `TimeoutError`, `ValidationError`. Покажи как определить кастомный тип ошибки в Go и как получатель может проверить конкретный тип через `errors.As`.

**Вопрос В.**
Ты накапливаешь все ошибки в слайс и выводишь в конце. В продакшне иногда нужна другая стратегия — **fail fast**: при первой ошибке остановить всё. Как ты модифицируешь свою программу чтобы реализовать fail fast — с использованием контекста и отмены?

## Критерии правильного решения

Твоё решение корректно если выполняются все условия: воркер не содержит ни одного `log` или `fmt` для ошибок — только передача через канал; `Result` всегда содержит исходный `Job` независимо от того успех это или ошибка; получатель проверяет `r.Err != nil` перед использованием `r.Value`; отправка в `results` защищена `select` с `ctx.Done()`; итоговая статистика показывает ровно 10 успешных и 5 ошибок; программа завершается чисто без дедлока.
