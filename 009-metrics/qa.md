## Блок 1: Атомарные операции — физика и механизм

**Q: Что происходит на уровне процессора когда выполняется `atomic.Int64.Add(1)`? Почему это быстрее чем `mu.Lock(); counter++; mu.Unlock()`?**

A: Атомарная операция `Add` транслируется в одну машинную инструкцию с префиксом `LOCK` — например `LOCK XADD` на x86. Префикс `LOCK` говорит процессору: "выполни следующую инструкцию атомарно, заблокировав шину памяти для других ядер на время этой операции". Это занимает порядка 5-10 наносекунд. Мьютекс работает иначе: `Lock()` требует системного вызова или спинлока, переключения контекста если мьютекс занят, записи в память для обозначения захвата. Это 50-100+ наносекунд. Принципиальное отличие: атомарная операция не может быть прервана между чтением и записью — это гарантирует процессор аппаратно. Мьютекс гарантирует это программно через ОС. Аппаратная гарантия всегда быстрее программной.

---

**Q: На 64-битной системе `int64` читается и записывается одной инструкцией — разве это не атомарно само по себе? Зачем нужен `atomic.Int64`?**

A: Одна инструкция чтения/записи не означает видимость изменений для других горутин. В многопроцессорной системе каждое ядро имеет свой кэш (L1, L2, L3). Когда горутина на ядре А записывает в `counter` — значение сначала попадает в кэш ядра А. Горутина на ядре Б может читать старое значение из своего кэша. Без явной синхронизации нет гарантии когда изменение станет видимым для Б. `atomic` операции включают memory barrier — барьер памяти — который заставляет процессор сбросить кэш и обеспечить видимость изменений для всех ядер. Модель памяти Go явно говорит: без синхронизирующего примитива (канал, мьютекс, atomic) нет гарантий порядка видимости операций между горутинами. Прямой `counter++` без atomic — это гонка данных даже если она "работает" на конкретном железе.

---

**Q: Запусти в голове этот тест — что выведет детектор гонок и почему:**

```go
var counter int64

go func() { counter++ }()
go func() { counter++ }()

time.Sleep(time.Second)
fmt.Println(counter)
```

A: Детектор гонок (`go run -race`) обнаружит data race и выведет предупреждение с указанием строк где происходят конкурентные записи без синхронизации. Программа может напечатать 1 или 2 — результат недетерминирован. Два сценария: первая горутина читает 0, вторая читает 0 (ещё не видит изменения первой), обе записывают 1 — итог 1 вместо 2. Или они выполняются последовательно — итог 2. На практике детектор гонок инструментирует каждый доступ к памяти и отслеживает happens-before отношения. Если два доступа к одной ячейке памяти не связаны happens-before и хотя бы один из них — запись, это data race. Исправление: `atomic.AddInt64(&counter, 1)` или `var counter atomic.Int64; counter.Add(1)`.

---

**Q: Чем `atomic.Int64` (Go 1.19+) отличается от старых функций `atomic.AddInt64`, `atomic.LoadInt64`?**

A: Старый API — пакетные функции которые принимают указатель: `atomic.AddInt64(&counter, 1)`, `atomic.LoadInt64(&counter)`. Новый API — методы на структуре: `counter.Add(1)`, `counter.Load()`. Новый API лучше по нескольким причинам. Первая — невозможно случайно забыть взять адрес или передать не тот тип: методы работают на конкретной структуре. Вторая — `atomic.Int64` нельзя копировать (содержит внутри `noCopy` маркер который `go vet` отслеживает) — это предотвращает класс ошибок когда счётчик копируется и операции идут на разные экземпляры. Третья — код читается естественнее: `p.metrics.Processed.Add(1)` vs `atomic.AddInt64(&p.metrics.Processed, 1)`.

## Блок 2: Два типа метрик — структура дизайна

**Q: Почему в коде два типа: `metrics` (строчная, с `atomic.Int64`) и `Metrics` (заглавная, с `int64`)? Разве не проще хранить и возвращать один тип?**

A: `atomic.Int64` нельзя копировать — это задокументированное ограничение. Если бы `GetMetrics()` возвращал структуру с `atomic.Int64` полями, Go при возврате скопировал бы структуру — и `go vet` немедленно сообщил бы об ошибке. Поэтому нужны два разных типа: `metrics` живёт внутри `Pool` как разделяемое состояние которое обновляется атомарно из нескольких горутин, `Metrics` — это снимок (snapshot) для чтения, простая структура с обычными `int64` которую можно копировать, передавать, логировать. `GetMetrics()` делает явное преобразование через `Load()`: читает атомарное значение и помещает его в копируемую структуру. Это стандартный паттерн разделения "живого состояния" и "снимка для наблюдателя".

---

**Q: `GetMetrics()` возвращает снимок. Может ли наблюдатель получить несогласованное состояние — например `Processed` уже обновлён а `Errors` ещё нет для одной и той же задачи?**

A: Да, может. `GetMetrics()` делает четыре отдельных `Load()` — между ними воркеры продолжают обновлять счётчики. Строго говоря, снимок не атомарен относительно всех четырёх полей одновременно. Это называется eventual consistency на уровне снимка. Является ли это проблемой для мониторинга? Нет. Метрики для мониторинга не требуют строгой согласованности в каждый конкретный момент — они требуют корректности в агрегате. Дашборд показывает тренды, не точные значения в конкретную миллисекунду. Если нужна строгая согласованность снимка — нужен мьютекс вокруг всех четырёх обновлений и чтений. Но это убьёт производительность которую мы выиграли от atomic. Trade-off осознанный.

---

**Q: `QueueDepth()` возвращает `len(p.jobs)`. Это значение устаревает ещё до того как ты его используешь. Почему это нормально для метрики но неприемлемо для управляющей логики?**

A: `len()` для канала — атомарная операция встроенная в рантайм, но между вызовом `len` и любым следующим действием другая горутина может изменить состояние канала. Для метрики это нормально: наблюдатель хочет знать приблизительную заполненность очереди прямо сейчас — погрешность в несколько элементов не влияет на корректность мониторинга. Для управляющей логики это опасно: если написать `if len(p.jobs) == 0 { pool.Shutdown() }` — между проверкой и `Shutdown` другая горутина могла добавить задачи. `Shutdown` вызовется когда задачи ещё есть — они будут потеряны. Правило: `len(channel)` можно использовать для наблюдения и эвристик, нельзя для принятия критических решений о состоянии системы.

## Блок 3: Порядок defer и метрики

**Q: В `executeTask` два `defer` обновляют метрики. Назови точный порядок их выполнения при успехе, ошибке и панике — и объясни почему порядок критически важен.**

A: LIFO — последний зарегистрированный выполняется первым.

При **успехе**: второй `defer` (Processed++, InFlight--, recover→nil) выполняется первым. Первый `defer` (проверка Errors/Panic) выполняется вторым — `result.Err == nil`, ничего не инкрементируется.

При **ошибке**: второй `defer` выполняется первым — Processed++, InFlight--, recover→nil. Первый `defer` — `result.Err != nil` и не `PanicError` → Errors++.

При **панике**: паника в `ProcessFn` → второй `defer` выполняется — Processed++, InFlight--, `recover()` ловит панику, устанавливает `result.Err = &PanicError{...}`. Первый `defer` — `result.Err != nil` и `errors.As → PanicError` → Panic++.

Порядок критичен потому что первый `defer` зависит от значения `result.Err` которое устанавливается вторым `defer` через `recover`. Если бы они выполнялись в прямом порядке — при панике первый `defer` прочитал бы `result.Err == nil` (recover ещё не отработал) и не инкрементировал бы `Panic`. Метрика была бы неверной.

---

**Q: `InFlight` инкрементируется ДО регистрации `defer` через `p.metrics.InFlight.Add(1)` в начале функции а декрементируется внутри `defer`. Почему именно такой порядок и что произойдёт если поставить `InFlight.Add(1)` тоже в `defer`?**

A: `InFlight` означает "задача прямо сейчас в обработке". Инкремент должен произойти в момент начала обработки — до любой возможной паники или ошибки. Декремент через `defer` гарантирует что он произойдёт при любом исходе включая панику. Если поставить `InFlight.Add(1)` в третий `defer` в начале функции — он выполнился бы последним при выходе, уже после декремента в другом `defer`. Получилось бы: задача завершилась → InFlight-- → InFlight++ → итог InFlight не изменился. Метрика была бы бессмысленной. Правило простое: начало работы фиксируется в начале функции, конец работы фиксируется через `defer`.

## Блок 4: Горутина-наблюдатель и паттерн завершения

**Q: Горутина-наблюдатель читает метрики каждые 50ms. Почему она должна завершаться корректно через сигнальный канал, а не просто "умереть" при завершении `main`?**

A: Когда `main` завершается — Go убивает все горутины принудительно, без выполнения их `defer`. Если горутина-наблюдатель держит ресурсы (тикер, открытые соединения, незакрытые файлы) — они не освободятся через `defer ticker.Stop()` потому что `defer` не выполнится. В данном случае незакрытый тикер — это утечка ресурса: внутренний таймер продолжает тикать до принудительного убийства. Кроме того горутина-наблюдатель может не успеть вывести финальный снимок метрик. Корректное завершение через сигнальный канал позволяет горутине выполнить `defer ticker.Stop()` и любую финализирующую логику — это чистое завершение. Это та же философия что и у `Shutdown` vs принудительного убийства процесса.

---

**Q: Почему `done chan struct{}` с буфером 1 правильнее чем небуферизованный `done chan struct{}`?**

A: Небуферизованный канал требует что отправитель и получатель встретились одновременно — рандеву. Если наблюдатель уже завершился (например по таймауту контекста) до того как надсмотрщик пытается отправить в `done` — надсмотрщик заблокируется навсегда. Это утечка горутины-надсмотрщика. С буфером 1: надсмотрщик кладёт `struct{}{}` в буфер и немедленно продолжает — независимо от того читает ли наблюдатель. Семантика здесь "выстрели и забудь": надсмотрщик не должен ждать подтверждения что наблюдатель получил сигнал. Буфер 1 — минимальный буфер который обеспечивает эту семантику. Буфер больше 1 не нужен — сигнал отправляется ровно один раз.

---

**Q: Почему нельзя использовать `pool.Results()` как сигнал завершения для горутины-наблюдателя вместо отдельного `done` канала?**

A: `pool.Results()` возвращает канал результатов который читает `main` через `for range`. Если наблюдатель тоже читает из этого канала в `select` — он конкурирует с `main` за результаты. Go при наличии нескольких читателей отдаёт значение случайному из них — часть результатов уйдёт наблюдателю вместо `main`. Данные будут потеряны с точки зрения бизнес-логики. Это классическая ошибка "два читателя одного канала данных". Сигнальный канал `done` и канал данных `results` должны быть разделены — у них разная семантика и разные получатели.

## Блок 5: Бенчмарк atomic vs mutex

**Q: Ты написал бенчмарк atomic vs mutex. Типичный результат: atomic ~8ns, mutex ~60ns. Объясни откуда берётся разница в 7-8 раз на горячем пути.**

A: Разница складывается из нескольких компонентов. Atomic: одна инструкция `LOCK XADD` — блокировка шины памяти на одну операцию, без системных вызовов, без переключения контекста, без очереди ожидания. Это ~5-10ns. Mutex при отсутствии конкуренции: `Lock()` проверяет атомарный флаг (занят/свободен) — если свободен, захватывает через CAS операцию. Это ~20-30ns — уже дороже atomic потому что две операции вместо одной. Mutex при конкуренции (несколько горутин одновременно): проигравшая горутина переходит в состояние ожидания — рантайм снимает её с P, ставит в очередь мьютекса, при освобождении пробуждает. Это ~50-100ns и более из-за переключения контекста. На горячем пути с 8 конкурентными горутинами мьютекс почти всегда в состоянии конкуренции — отсюда ~60ns против ~8ns для atomic.

**Q: `CompareAndSwap` (CAS) — атомарная операция которой не было в простом примере. Когда простого `Add` недостаточно и нужен CAS?**

A: `Add` безусловно прибавляет значение — он не проверяет текущее значение перед изменением. Это работает для счётчиков. Но если нужно атомарно перевести переменную из одного состояния в другое только если она в ожидаемом состоянии — нужен CAS. Пример: атомарная смена состояния пула из "running" в "stopped":

```go
const (
    stateRunning int32 = 0
    stateStopped int32 = 1
)

var state atomic.Int32

// Только одна горутина успешно переведёт состояние
// Остальные получат false — состояние уже изменено
if state.CompareAndSwap(stateRunning, stateStopped) {
    // мы первые — выполняем финализацию
    close(p.jobs)
}
```

Если использовать `Add` для этого — две горутины могут одновременно "победить": обе читают 0 (running), обе прибавляют 1, обе думают что они первые. CAS гарантирует что только одна горотина успешно изменит состояние — атомарно читает, сравнивает и записывает за одну неделимую операцию.

## Блок 6: Паттерны наблюдаемости

**Q: Throughput как производная метрика: как вычислить задач-в-секунду из двух снимков `Metrics` без дополнительных атомарных операций?**

A: Через разницу значений `Processed` между двумя снимками:

```go
prev := pool.GetMetrics()
time.Sleep(50 * time.Millisecond)
curr := pool.GetMetrics()

delta := curr.Processed - prev.Processed
throughput := float64(delta) / 0.050 // задач в секунду
```

Никаких дополнительных атомарных операций — просто арифметика над двумя int64. Это стандартный паттерн для вычисления rate метрик в мониторинге: Prometheus делает то же самое вычисляя `rate()` как разность значений counter за период. Важно что `Processed` — монотонно возрастающий counter (никогда не уменьшается), что делает вычисление delta всегда корректным.

**Q: Финальные метрики показали `processed: 100`, `errors: 27`, `panics: 20`, `inflight: 0`. Проверь корректность: сколько успешных задач? Сходится ли с теоретическим расчётом для задач 1-100?**

A: Успешных: 100 - 27 - 20 = 53. Теоретический расчёт: числа делящиеся на 5 из 1-100: 5,10,15,...,100 = 20 паник. Числа делящиеся на 3 но не на 5 из 1-100: делящихся на 3 всего 33, делящихся на 15 (на 3 и на 5) = 6. Итого ошибок: 33 - 6 = 27. Успешных: 100 - 20 - 27 = 53. Метрики полностью совпадают с теоретическим расчётом — это подтверждает что ни одна задача не потеряна и все счётчики работают корректно. Именно такой детерминированный контрольный расчёт — правильный способ верифицировать конкурентный код.
