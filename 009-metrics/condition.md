# 009 — Metrics: Атомарные счётчики без мьютексов

## Контекст

К этому моменту твой пул умеет принимать задачи, обрабатывать их, возвращать результаты, переживать паники, корректно завершаться и управлять нагрузкой через backpressure. Но у него есть слепое пятно: ты не знаешь что происходит внутри прямо сейчас. Сколько задач обработано? Сколько упало с ошибкой? Сколько задач в обработке прямо сейчас? Без ответов на эти вопросы пул — чёрный ящик.

В продакшне наблюдаемость (observability) — это не опция, это требование. Prometheus, Grafana, алерты, дашборды — всё это строится на метриках которые сервис собирает о себе сам. Каждый воркер должен атомарно обновлять счётчики при каждой операции — и делать это **без мьютексов**, потому что мьютекс на каждый инкремент счётчика в горячем пути — это узкое место.

Здесь появляется `sync/atomic` — пакет для атомарных операций на уровне процессора. Атомарная операция выполняется как единое неделимое действие: никакая другая горутина не может увидеть промежуточное состояние. Это быстрее мьютекса потому что не требует системного вызова и переключения контекста — только одна инструкция процессора с префиксом `LOCK`.

В этой задаче ты добавишь в пул живые метрики и научишься читать их из любого места без блокировок.

---

## Задача

Расширь структуру `Pool` из задачи 008. Добавь вложенную структуру `Metrics` и метод для её получения.

Структура `Metrics` содержит четыре поля: `Processed atomic.Int64` — общее число задач завершённых воркером (успешно или с ошибкой, но не паника отдельно — всё что прошло через `executeTask`); `Errors atomic.Int64` — число задач завершившихся с ненулевым `Err`; `Panics atomic.Int64` — число пойманных паник; `InFlight atomic.Int64` — число задач находящихся в обработке прямо сейчас (увеличивается при взятии задачи, уменьшается при завершении).

Метод `QueueDepth() int` на структуре `Pool` возвращает текущую длину канала `jobs` — сколько задач ожидают обработки.

Метод `GetMetrics() Metrics` возвращает **снимок** метрик в момент вызова. Это не указатель на живую структуру — это копия значений на данный момент. Подумай почему снимок лучше чем прямой доступ к структуре.

В `main` запусти горячий пул с 4 воркерами и 100 задачами. Параллельно запусти горутину-наблюдатель которая каждые `50ms` печатает текущие метрики. Наблюдатель завершается когда пул завершает работу.

---

## Требования

- Все четыре поля `Metrics` используют `atomic.Int64` из пакета `sync/atomic` — не `int64` с мьютексом.
- `InFlight` точно отражает число задач в обработке: инкрементируется **до** начала `process`, декрементируется через `defer` — чтобы учесть и панику.
- `QueueDepth` возвращает `len(p.jobs)` — это атомарная операция встроенная в рантайм, дополнительная защита не нужна.
- Горутина-наблюдатель читает метрики через `GetMetrics()` — не напрямую к полям структуры.
- После завершения всех задач финальные метрики должны показать: `InFlight == 0`, `Processed == 100`, `Errors + Panics <= 100`.
- Детектор гонок `go run -race .` молчит.

## Вопросы ДО написания кода

**Вопрос 1.**
В чём физическая разница между `atomic.Int64` и `int64` с `sync.Mutex`? Оба обеспечивают безопасный конкурентный доступ — но через разные механизмы. Что происходит на уровне процессора когда выполняется `atomic.Add`? Почему это быстрее чем `mu.Lock(); counter++; mu.Unlock()`?

**Вопрос 2.**
Рассмотри этот код и найди гонку данных:

```go
type Metrics struct {
    Processed int64
    Errors    int64
}

// в воркере:
m.Processed++ // не атомарно
```

На 64-битной системе `int64` читается и записывается одной инструкцией — разве это не атомарно само по себе? Почему всё равно нужен `atomic`? Подсказка: подумай про кеши процессора и модель памяти Go.

**Вопрос 3.**
`InFlight` должен увеличиваться до начала обработки и уменьшаться после — в том числе при панике. Как `defer` помогает гарантировать декремент при любом исходе выполнения задачи? Напиши в голове порядок выполнения инструкций в `executeTask` включая `defer`.

**Вопрос 4.**
`GetMetrics()` возвращает копию структуры `Metrics` а не указатель. Но `atomic.Int64` нельзя копировать — это написано в документации. Как тогда вернуть снимок метрик? Какой тип нужен для снимка чтобы он содержал `int64` а не `atomic.Int64`?

**Вопрос 5.**
Горутина-наблюдатель читает метрики каждые `50ms`. Между двумя чтениями воркеры продолжают обновлять счётчики. Может ли наблюдатель прочитать **несогласованное** состояние — например `Processed` уже обновлён а `Errors` ещё нет для одной и той же задачи? Является ли это проблемой для мониторинга? Как называется этот тип несогласованности?

**Вопрос 6.**
`len(p.jobs)` возвращает текущую длину буферизованного канала. Это значение уже устарело к моменту когда ты его используешь — между `len` и печатью другая горутина могла добавить или убрать задачу. Почему это нормально для метрики `QueueDepth` но было бы неприемлемо для принятия управляющих решений (например "если очередь пуста — завершись")?

## Эксперимент: мьютекс vs atomic под нагрузкой

Это обязательная часть задачи — выполни до написания финальной версии.

Напиши бенчмарк сравнивающий два подхода к инкременту счётчика из 8 конкурентных горутин, каждая делает 1 000 000 инкрементов:

```go
// Подход А: мьютекс
var mu sync.Mutex
var counter int64
mu.Lock()
counter++
mu.Unlock()

// Подход Б: atomic
var counter atomic.Int64
counter.Add(1)
```

Запусти через `go test -bench=. -benchmem`. Зафиксируй разницу в наносекундах на операцию. Типичный результат: atomic быстрее в 3-10 раз. Запиши результаты в комментарии в коде.

## Эксперименты после решения

**Эксперимент А.** Запусти с `-race`. Детектор должен молчать. Теперь замени одно поле `atomic.Int64` на обычный `int64` и напрямую инкрементируй его без atomic: `p.metrics.Processed++`. Запусти с `-race` снова. Детектор должен обнаружить гонку. Зафиксируй что именно он сообщает.

**Эксперимент Б.** Добавь в наблюдатель подсчёт **скорости обработки**: сколько задач обработано за последние 50ms (дельта между двумя снимками). Это позволяет видеть throughput в реальном времени — задач в секунду. Реализуй без дополнительных атомарных операций — только через разницу снимков.

**Эксперимент В.** Увеличь число воркеров с 4 до 32 при тех же 100 задачах. Как изменяется `InFlight` в выводе наблюдателя? Почему `InFlight` никогда не превышает число задач в очереди плюс число воркеров?

## Вопросы ПОСЛЕ решения

**Вопрос А.**
`atomic.Int64` в Go 1.19+ — это структура с методами `Load`, `Store`, `Add`, `Swap`, `CompareAndSwap`. До Go 1.19 использовались функции `atomic.AddInt64`, `atomic.LoadInt64` и т.д. Чем новый API лучше старого? Что мешало сделать `atomic.Int64` копируемым — ведь это просто обёртка над `int64`?

**Вопрос Б.**
`CompareAndSwap` (CAS) — это атомарная операция которая обновляет значение только если оно равно ожидаемому. Это примитив для построения lock-free структур данных. Приведи пример где CAS необходим — где простого `Add` недостаточно. Подсказка: подумай про атомарный переход состояния пула из "running" в "stopped".

**Вопрос В.**
Метрики которые ты собираешь — это внутренние метрики Go процесса. В продакшне их экспортируют в Prometheus через `/metrics` endpoint. Пакет `expvar` из стандартной библиотеки Go предоставляет готовый механизм для этого. Как `expvar.Int` соотносится с `atomic.Int64` — они решают одну задачу или разные?

## Критерии правильного решения

Твоё решение корректно если выполняются все условия одновременно: все четыре поля `Metrics` используют `atomic.Int64`; `InFlight` точно инкрементируется до обработки и декрементируется через `defer`; `GetMetrics()` возвращает снимок с типом `int64` а не `atomic.Int64`; горутина-наблюдатель корректно завершается после завершения пула; финальные метрики показывают `InFlight == 0` и `Processed == 100`; детектор гонок молчит; бенчмарк написан и результаты зафиксированы.
