# 006 — Panic Recovery: Защита воркера от смерти

## Контекст

В задаче 005 ты научил воркера передавать ошибки как данные. Но `error` — это контролируемая ситуация: функция знает что что-то пошло не так и явно возвращает ошибку. Паника — это другое. Паника это неожиданный, неконтролируемый сбой: обращение к nil указателю, выход за границы слайса, деление на ноль, явный вызов `panic(...)` в чужом коде который ты не контролируешь.

По умолчанию паника в горутине **убивает весь процесс**. Не только эту горутину — весь процесс целиком. В контексте worker pool это означает что одна сломанная задача может убить все остальные воркеры и уронить сервис. Это неприемлемо в продакшне.

Решение — `recover`. Это встроенная функция которая перехватывает панику и возвращает управление в точку после `defer`. Но у `recover` есть жёсткое ограничение которое новички часто нарушают: он работает **только внутри функции вызванной через `defer`** и **только в той же горутине** где произошла паника. Именно это ограничение делает правильную реализацию нетривиальной.

После этой задачи воркер станет по-настоящему надёжным: он выживает после паники, продолжает работу, и честно сообщает о произошедшем через тот же канал результатов.

---

## Задача

Напиши программу в которой:

1. Функция `process(job Job) (value int, err error)` — работает как в задаче 005, но добавь новое условие: если `job.ID == 5` — вызывает `panic("unexpected nil pointer in job 5")`. Остальная логика та же: ошибка если ID делится на 3, иначе квадрат значения.

2. Функция `executeTask(workerID int, job Job) (result Result)` — **отдельная функция** (не анонимная) которая вызывает `process` и содержит `recover` через `defer`. Если паника поймана — возвращает `Result{Job: job, Err: fmt.Errorf("worker %d panic on job %d: %v", workerID, job.ID, recovered), WorkerID: workerID}`. Если паники не было — возвращает нормальный результат.

3. Функция `startWorker` — использует `executeTask` для каждой задачи. После паники воркер **продолжает работу** — берёт следующую задачу из канала.

4. В `main` — отправь 10 задач с ID от 1 до 10. Задача с ID 5 вызовет панику. Запусти 3 воркера. Собери все 10 результатов. Напечатай статистику: успешные, ошибки (от process), паники (от recover).

---

## Требования

- `recover` находится **только в `executeTask`** — не в `startWorker` и не в анонимной функции внутри воркера.
- После паники воркер продолжает работу — обрабатывает оставшиеся задачи.
- Паника превращается в `Result` с заполненным `Err` — через тот же механизм что и обычные ошибки.
- Программа не падает — процесс остаётся живым даже когда `process` вызывает `panic`.
- Итог: 7 успешных (ID: 1,2,4,7,8,10 — не делятся на 3 и не равны 5), 2 ошибки (ID: 3, 9 — делятся на 3), 1 паника (ID: 5). Проверь арифметику перед написанием.

## Вопросы ДО написания кода

**Вопрос 1.**
Почему паника в горутине убивает весь процесс, а не только эту горутину? Ведь горутины изолированы — у каждой свой стек. В чём логика такого дизайна? Чем это отличается от поведения потоков в других языках (например Java где uncaught exception убивает только поток)?

**Вопрос 2.**
Рассмотри три варианта размещения `recover` и объясни какой работает и почему остальные нет:

```go
// Вариант А
func executeTask(job Job) (result Result) {
    defer func() {
        if r := recover(); r != nil {
            result = Result{Err: fmt.Errorf("panic: %v", r)}
        }
    }()
    value, err := process(job)
    result = Result{Value: value, Err: err}
    return
}

// Вариант Б
func executeTask(job Job) Result {
    if r := recover(); r != nil { // recover вне defer
        return Result{Err: fmt.Errorf("panic: %v", r)}
    }
    value, err := process(job)
    return Result{Value: value, Err: err}
}

// Вариант В
func startWorker(...) {
    defer func() {
        if r := recover(); r != nil { // recover в воркере, не в executeTask
            fmt.Println("recovered:", r)
        }
    }()
    for job := range jobs {
        executeTask(job) // паника здесь
    }
}
```

Вариант А работает. Почему варианты Б и В не дают нужного результата даже если компилируются?

**Вопрос 3.**
В варианте А используется именованное возвращаемое значение `(result Result)`. Почему это важно для корректной работы `recover`? Что произойдёт если убрать именованный возврат и попробовать вернуть значение из `defer`?

**Вопрос 4.**
`recover()` возвращает `any` (то есть `interface{}`). Паника может быть вызвана с любым значением: `panic("string")`, `panic(42)`, `panic(errors.New("err"))`, `panic(nil)`. Как правильно превратить это значение в `error`? Напиши в голове функцию `toError(r any) error` которая обрабатывает все эти случаи корректно, включая `panic(nil)`.

**Вопрос 5.**
После того как `recover` поймал панику — воркер продолжает работу. Это означает что он снова войдёт в `select` и возьмёт следующую задачу. Что если следующая задача тоже паникует? И следующая? Воркер выживает после каждой паники независимо — почему? Нарисуй в голове стек вызовов для второй паники.

**Вопрос 6.**
`recover` работает только в **той же горутине** где произошла паника. Это означает что нельзя поймать панику из другой горутины. Рассмотри этот код и объясни почему `recover` не поймает панику:

```go
func startWorker() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("caught:", r) // это никогда не выполнится
        }
    }()

    go func() {
        panic("panic in nested goroutine") // другая горутина — recover не поможет
    }()

    time.Sleep(time.Second)
}
```

Что это означает для дизайна воркера — где должен находиться `recover` относительно горутины которая может запаниковать?

## Эксперимент: смерть воркера без recover

Это обязательная часть задачи.

Сначала напиши версию воркера **без `recover`**. Просто вызывай `process` напрямую без какой-либо защиты. Запусти программу с задачей ID=5 которая паникует.

Наблюдай вывод: Go напечатает стек трейс паники и завершит **весь процесс**. Запиши что именно выводит рантайм — сколько строк стек трейса, какие горутины упоминаются, что происходит с задачами которые ещё не были обработаны.

Потом добавь `recover` в `executeTask` и убедись что программа теперь выживает и обрабатывает все 10 задач.

Это не просто эксперимент — это прямое доказательство зачем нужен `recover` в продакшн воркере.

## Эксперименты после решения

**Эксперимент А.** Добавь `panic(nil)` как отдельный кейс (например для job.ID == 7). Поймает ли твой `recover` эту панику? Что вернёт `recover()` в этом случае? Как отличить `panic(nil)` от отсутствия паники? Подсказка: в Go 1.21 поведение `panic(nil)` изменилось — изучи это изменение.

**Эксперимент Б.** Намеренно поставь `recover` в `startWorker` через `defer` — вне `executeTask`. Что произойдёт когда сработает паника? Воркер продолжит работу или завершится? Почему поведение отличается от правильного варианта?

**Эксперимент В.** Запусти программу с флагом `-race`: `go run -race .`. Есть ли race conditions в твоём решении? Если `recover` поймал панику и записал в именованный возврат — это происходит в одной горутине, не нужен мьютекс. Убедись что детектор гонок молчит.

## Вопросы ПОСЛЕ решения

**Вопрос А.**
Паника и `recover` — это не замена `error`. В каких ситуациях уместно использовать `panic` в продакшн коде? Назови конкретные примеры из стандартной библиотеки Go где `panic` используется намеренно — и объясни логику этого выбора.

**Вопрос Б.**
В `executeTask` ты используешь именованный возврат и модифицируешь его в `defer`. Это паттерн называется "named return with defer mutation". Покажи как этот же паттерн используется в стандартной библиотеке для реализации транзакций: функция должна сделать `rollback` при панике и `commit` при успехе.

**Вопрос В.**
После `recover` у тебя есть значение типа `any`. В реальном коде паники могут содержать структурированную информацию: стек трейс, контекст запроса, ID пользователя. Как получить стек трейс внутри `recover` используя только стандартную библиотеку? Какой пакет и какая функция нужны?

## Критерии правильного решения

Твоё решение корректно если выполняются все условия одновременно: `recover` находится только в `executeTask` — не в воркере, не в анонимной функции в цикле; после паники на задаче ID=5 воркер продолжает обрабатывать оставшиеся задачи; паника превращается в `Result` с `Err` — через тот же канал что и обычные ошибки; итоговая статистика: 7 успешных, 2 ошибки, 1 паника, итого 10; программа не падает — процесс остаётся живым; детектор гонок `go run -race .` молчит.
