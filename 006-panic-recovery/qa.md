# QA — 006 Panic Recovery: Защита воркера от смерти

Этот файл строится поверх QA-001 — QA-005. Здесь только новое что появилось в задаче 006: паника как механизм, recover, именованный возврат, архитектурная граница между инфраструктурой и бизнес-логикой. Отвечай вслух до чтения ответа.

---

## Блок 1: Паника — что это и почему она убивает процесс

**Q: Чем паника отличается от обычной ошибки в Go?**

A: Обычная ошибка — это ожидаемый исход который функция явно возвращает через `(value, error)`. Вызывающий код знает что ошибка возможна и обязан её проверить. Паника — это неожиданный, неконтролируемый сбой который прерывает нормальный поток выполнения: обращение к nil указателю, выход за границы слайса, деление на ноль, явный вызов `panic(...)`. Паника не возвращается через return — она разворачивает стек вызовов вверх, выполняя все отложенные функции `defer` по пути, пока не дойдёт до вершины горутины. Если никто не поймал панику через `recover` — горутина завершается аварийно и рантайм убивает весь процесс. Это принципиальное отличие от ошибки: ошибку можно проигнорировать (хотя не стоит), панику проигнорировать нельзя.

---

**Q: Почему паника в одной горутине убивает весь процесс, а не только эту горутину? Ведь горутины изолированы — у каждой свой стек.**

A: Изоляция стека означает что горутины не делят локальные переменные — но это не означает изоляцию от отказа. Философия Go здесь намеренно жёсткая: непойманная паника считается признаком того что программа находится в неопределённом, потенциально опасном состоянии. Продолжать работу в таком состоянии — значит рисковать повреждением данных или некорректными результатами. Поэтому рантайм принимает консервативное решение: если программист не предусмотрел обработку этой ситуации — лучше упасть чисто чем продолжать работать неверно. Это отличается от Java где непойманное исключение в потоке убивает только этот поток — Go считает что такое "тихое" завершение потока хуже явного падения всего процесса.

---

**Q: Что именно выводит рантайм Go при непойманной панике и что из этого можно узнать?**

A: Рантайм выводит сообщение `goroutine N [running]: panic: <значение>` а затем полный стектрейс горутины где произошла паника — от точки паники до верхушки горутины. Затем выводятся стектрейсы всех остальных горутин которые были живы в момент паники — с их состоянием (`running`, `chan receive`, `sleep` и т.д.) и строками кода где они заблокированы. Из этого вывода можно узнать точную строку где произошла паника, всю цепочку вызовов которая к ней привела, и состояние всей программы в момент сбоя. Это полная посмертная диагностика — в отличие от логов которые могут не содержать нужного контекста.

---

**Q: Чем `fatal error` от дедлока отличается от паники и почему `recover` не помогает в обоих случаях для `fatal error`?**

A: Паника запускается через `panic(value)` и может быть поймана через `recover()` в `defer` — это управляемый механизм обработки исключительных ситуаций. `fatal error` — это диагностика самого рантайма Go о ситуациях которые он считает неустранимыми: дедлок (`all goroutines are asleep`), конкурентная запись в map (`concurrent map writes`), отрицательный счётчик WaitGroup. Рантайм не запускает механизм паники для этих ситуаций — он напрямую завершает процесс через внутренний `throw`. Поэтому `recover` их не ловит: `recover` работает только внутри механизма паники, а `fatal error` этот механизм обходит.

## Блок 2: recover — механизм и ограничения

**Q: Где именно должен находиться `recover` чтобы поймать панику, и почему это ограничение существует?**

A: `recover` работает только внутри функции вызванной через `defer`, и только в той горутине где произошла паника. Это два жёстких ограничения. Первое ограничение существует потому что `recover` должен перехватить управление в момент разворачивания стека — а это происходит только внутри механизма `defer`. Вызов `recover()` вне `defer` просто возвращает `nil` и не имеет никакого эффекта на панику. Второе ограничение существует потому что паника разворачивает стек конкретной горутины — другая горутина имеет свой независимый стек и физически не может вмешаться в этот процесс. Поэтому нельзя написать "глобальный обработчик паник" который ловит паники из всех горутин — каждая горутина должна защищаться сама.

---

**Q: Рассмотри три варианта размещения `recover` и объясни какой работает и почему остальные нет:**

```go
// Вариант А
func executeTask(job Job) (result Result) {
defer func() {
if r := recover(); r != nil {
result.Err = fmt.Errorf("panic: %v", r)
}
}()
value, err := process(job)
result = Result{Value: value, Err: err}
return
}

// Вариант Б
func executeTask(job Job) Result {
if r := recover(); r != nil { // recover вне defer
return Result{Err: fmt.Errorf("panic: %v", r)}
}
value, err := process(job)
return Result{Value: value, Err: err}
}

// Вариант В
func startWorker(...) {
defer func() {
if r := recover(); r != nil {
fmt.Println("recovered:", r)
// воркер завершился — не продолжает работу
}
}()
for job := range jobs {
executeTask(job) // паника здесь
}
}
```

A: Работает только вариант А. Вариант Б не работает потому что `recover()` вызывается вне `defer` — это просто вызов функции которая возвращает `nil`. Когда `process` паникует, рантайм начинает разворачивать стек и не возвращается к строке с `recover` — она уже была выполнена раньше и никакого эффекта не имела. Вариант В не даёт нужного результата по другой причине: `recover` стоит в `startWorker`, а не в `executeTask`. Когда паника произойдёт, она развернёт стек через `executeTask` вверх до `startWorker` где и будет поймана. Но в этот момент `executeTask` уже завершилась аварийно — нет возможности вернуть корректный `Result`. Воркер поймает панику, напечатает сообщение и... завершится, потому что после `recover` в `startWorker` цикл `for job := range jobs` уже не продолжится — функция вышла из defer и завершает выполнение. Воркер умрёт тихо вместо того чтобы продолжить обработку задач.

---

**Q: Что вернёт `recover()` если паники не было?**

A: Вернёт `nil`. Это важно для написания корректного кода с `recover`: всегда проверяй `if r := recover(); r != nil` — если паники не было, `r` равен `nil` и никакой специальной обработки не нужно. Именно эта проверка позволяет одному `defer` безопасно обрабатывать оба случая: нормальное завершение (ничего не делать) и паника (вернуть ошибку).

---

**Q: `recover()` возвращает `any`. Паника может быть вызвана с любым значением: `panic("string")`, `panic(42)`, `panic(errors.New("err"))`, `panic(nil)`. Как правильно превратить это в `error`?**

A: Нужно обработать разные типы явно. Наиболее надёжный подход — проверить является ли значение уже `error`, и если нет — обернуть его в строку:

```go
defer func() {
if r := recover(); r != nil {
switch v := r.(type) {
case error:
result.Err = fmt.Errorf("panic with error: %w", v)
default:
result.Err = fmt.Errorf("panic with value: %v", v)
}
}
}()
```

Особый случай — `panic(nil)`. До Go 1.21 `recover()` возвращал `nil` при `panic(nil)`, что было неотличимо от "паники не было". С Go 1.21 это исправлено: `panic(nil)` теперь оборачивается в специальный тип `*runtime.PanicNilError` и `recover()` возвращает ненулевое значение. Если ты поддерживаешь код на Go < 1.21, проверить `panic(nil)` программно было невозможно — это важное историческое ограничение.

## Блок 3: Именованный возврат и defer

**Q: Почему `executeTask` использует именованный возврат `(result Result)` и что изменится если убрать именование?**

A: Именованный возврат создаёт переменную `result` которая живёт на протяжении всего выполнения функции — включая выполнение `defer`. Функция в `defer` может читать и модифицировать эту переменную, и изменения будут отражены в возвращаемом значении. Без именованного возврата:

```go
// Без именованного возврата — не работает
func executeTask(job Job) Result {
defer func() {
if r := recover(); r != nil {
// здесь нет доступа к возвращаемому значению
// мы не можем вернуть Result{Err: ...} из defer
}
}()
// ...
}
```

Когда паника разворачивает стек, нормальный `return` не выполняется — функция не доходит до него. `defer` выполняется, но без именованного возврата нет способа передать значение из `defer` в вызывающий код. С именованным возвратом `defer` просто записывает в уже существующую переменную `result` и при завершении функции эта переменная возвращается автоматически. Это единственный механизм в Go который позволяет превратить панику в значение.

---

**Q: В каком порядке выполняются инструкции в `executeTask` при нормальном завершении и при панике?**

A: При нормальном завершении порядок такой: регистрируется `defer` с `recover`, выполняется `process(job)` возвращая `(value, err)`, формируется `result`, выполняется `return result`, после чего перед фактическим возвратом выполняется `defer` — `recover()` возвращает `nil`, условие `r != nil` ложно, `defer` завершается без изменений, функция возвращает `result`.

При панике порядок другой: регистрируется `defer` с `recover`, начинает выполняться `process(job)` и вызывает `panic(...)`, рантайм начинает разворачивать стек — `result` ещё не был присвоен (или был присвоен частично), выполняется `defer` — `recover()` перехватывает панику и возвращает значение переданное в `panic()`, условие `r != nil` истинно, `result.Err` устанавливается в `PanicError`, `defer` завершается, функция возвращает `result` с заполненным `Err` и нулевым `Value`.

---

**Q: Паттерн "named return with defer mutation" используется в стандартной библиотеке. Приведи пример где он критически необходим.**

A: Классический пример — транзакции в базе данных. Функция должна сделать `rollback` при любом сбое (включая панику) и `commit` только при успехе:

```go
func withTransaction(db *sql.DB, fn func(*sql.Tx) error) (err error) {
tx, err := db.Begin()
if err != nil {
return err
}

defer func() {
if p := recover(); p != nil {
// паника — откатываем транзакцию и пробрасываем панику
_ = tx.Rollback()
panic(p) // пробрасываем панику дальше
} else if err != nil {
// ошибка — откатываем транзакцию
_ = tx.Rollback()
} else {
// успех — фиксируем транзакцию
// err — именованный возврат, commit может вернуть ошибку
err = tx.Commit()
}
}()

err = fn(tx)
return err
}
```

Без именованного возврата `err` нельзя было бы проверить в `defer` — там не было бы доступа к значению которое возвращает `fn(tx)`. И нельзя было бы присвоить ошибку от `tx.Commit()` в возвращаемое значение.

## Блок 4: Архитектурная граница

**Q: Почему `recover` должен находиться в `executeTask`, а не в `startWorker` или в `process`?**

A: Ответ раскрывается через три роли этих функций. `process` — чистая бизнес-логика: она знает только как вычислить результат. Добавлять в неё `recover` значит давать бизнес-функции инфраструктурную ответственность, и тогда вызывающий код не сможет отличить "нормальную ошибку" от "пойманной паники" без парсинга строк. `startWorker` — управляет жизненным циклом горутины: читает из канала, управляет контекстом, отправляет результаты. Если поставить `recover` здесь, он поймает панику но воркер не сможет продолжить цикл — после `recover` в `defer` функция завершается. `executeTask` — это граница между инфраструктурой и бизнес-логикой. Её единственная задача: безопасно вызвать `process` и гарантированно вернуть `Result` при любом исходе включая панику. `recover` принадлежит именно этому слою — он знает и как поймать панику, и как упаковать её в тип который понимает инфраструктура.

---

**Q: Почему определять тип результата через `strings.Contains(err.Error(), "panic")` плохо, и как `PanicError` как отдельный тип решает эту проблему?**

A: `strings.Contains` создаёт неявный контракт основанный на строке — если строка изменится, классификация сломается молча и без ошибки компилятора. Кроме того любая бизнес-ошибка которая содержит слово "panic" будет неправильно классифицирована. Это хрупкость основанная на совпадении текста вместо семантики типов. `PanicError` как отдельный тип решает это через систему типов Go: `errors.As(err, &panicErr)` проверяет точный тип в цепочке оборачивания — это компилируемая, рефакторинг-безопасная проверка. Если ты переименуешь тип — компилятор немедленно покажет все места которые нужно обновить. Если строка изменится — ничего не сломается. Это принцип "сделай невозможное невозможным": компилятор охраняет инвариант вместо программиста.

---

**Q: Воркер продолжает работу после паники в `process`. Как это достигается архитектурно?**

A: Именно потому что `recover` находится в `executeTask` а не в `startWorker`. Когда `process` паникует — разворачивание стека останавливается в `executeTask` благодаря `recover` в `defer`. `executeTask` возвращает `Result{Err: &PanicError{...}}` как обычное значение. С точки зрения `startWorker` ничего экстраординарного не произошло — он получил результат с ошибкой, отправил его в канал через защищённый `select` и вернулся к началу цикла `for`. Следующая итерация читает следующую задачу из канала. Паника была полностью изолирована внутри `executeTask` и не "вылетела" в `startWorker`. Это и есть смысл архитектурной границы: всё что происходит ниже `executeTask` — остаётся ниже `executeTask`.

## Блок 5: Когда паника уместна в продакшн коде

**Q: В каких ситуациях использование `panic` в продакшн коде оправдано?**

A: Паника уместна в трёх категориях ситуаций. Первая — невозможное состояние: когда программа оказалась в ситуации которая по логике кода не должна была быть достижима, и продолжение работы опасно. Стандартная библиотека делает это в `encoding/json` при попытке кодировать тип который не поддерживается — это программная ошибка которую нужно исправить, не runtime-условие. Вторая — инициализация: когда программа не может корректно запуститься. `log.Fatal` и явный `panic` в `init()` или `main()` сигнализируют что конфигурация сломана и запуск невозможен — лучше упасть сразу чем работать некорректно. Третья — нарушение контракта API: когда вызывающий код передал параметры которые явно противоречат документации. `regexp.MustCompile` паникует если регулярное выражение невалидно — это ошибка программиста, не пользователя. Общий принцип: паника для ситуаций где продолжение работы хуже немедленного падения, `error` для ситуаций которые можно и нужно обработать в рантайме.

---

**Q: Как получить стектрейс внутри `recover` используя только стандартную библиотеку?**

A: Через пакет `runtime/debug` и функцию `debug.Stack()` которая возвращает текущий стек вызовов как `[]byte`. Это позволяет сохранить диагностическую информацию в момент паники:

```go
import "runtime/debug"

defer func() {
if r := recover(); r != nil {
stack := debug.Stack()
result.Err = &PanicError{
WorkerId:  workerId,
Recovered: r,
Stack:     string(stack), // полный стектрейс в момент паники
}
}
}()
```

В продакшне это критически важно: строка паники говорит что произошло, стектрейс говорит где и через какую цепочку вызовов. Без стектрейса диагностика сводится к угадыванию. Именно поэтому большинство продакшн фреймворков для recover всегда захватывают стек — Gin, Echo и другие HTTP фреймворки делают это в своём middleware для восстановления после паники.

---

**Q: `sync.Once` используется в задаче 007 для защиты от двойного `close`. Но почему двойной `close` канала приводит именно к панике, а не к ошибке?**

A: Это намеренное дизайнерское решение Go. Двойное закрытие канала считается программной ошибкой — нарушением инварианта "закрывает канал только отправитель, и только один раз". Если бы это возвращало ошибку, программист мог бы её проигнорировать и продолжить работу в потенциально некорректном состоянии. Паника заставляет программиста явно обработать ситуацию через `recover` или, лучше, написать код который не допускает двойного закрытия. Это согласуется с общей философией Go: для программных ошибок (нарушений контрактов кода) паника уместна, для runtime-условий (сеть недоступна, файл не найден) — ошибка. Задача 007 покажет как `sync.Once` устраняет необходимость ловить эту панику полностью — просто не допуская двойного закрытия.
