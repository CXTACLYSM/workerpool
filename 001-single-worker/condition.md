# 001 — Single Worker

## Контекст

Это первый шаг к построению полноценного Worker Pool.  
Прежде чем управлять несколькими воркерами — нужно понять как работает один.  
Здесь ты впервые соединишь три концепта: горутину, канал и WaitGroup.

---

## Задача

Напиши программу в которой:

1. Есть функция `startWorker(id int, jobs <-chan string, wg *sync.WaitGroup)`.  
   Она запускается как горутина и читает строки из канала `jobs`.  
   На каждую строку печатает: `worker-<id> processing: <строка>`.  
   Когда канал закрыт — печатает `worker-<id> done` и завершается.

2. В `main`:
    - Создай небуферизованный канал `jobs`.
    - Запусти одного воркера с `id = 0`.
    - Отправь в канал 5 строк: `"task-0"`, `"task-1"`, ..., `"task-4"`.
    - Закрой канал.
    - Дождись завершения воркера.
    - Напечатай `"main done"`.

---

## Требования

- Используй `range` для чтения из канала внутри воркера.
- Используй `sync.WaitGroup` для ожидания воркера в `main`.
- Программа не должна зависать и не должна завершаться раньше воркера.
- Никаких `time.Sleep` для синхронизации — только примитивы синхронизации.

---

## Ожидаемый вывод

```
worker-0 processing: task-0
worker-0 processing: task-1
worker-0 processing: task-2
worker-0 processing: task-3
worker-0 processing: task-4
worker-0 done
main done
```

> Порядок строк processing гарантирован в данном случае — почему?  
> Подумай об этом после того как напишешь код.

## Критерии правильного решения

Твоё решение корректно если:

- Программа завершается без зависания во всех вариантах (с закрытием канала и с ожиданием).
- Вывод соответствует ожидаемому.
- `wg.Add(1)` вызывается **до** `go`, а не внутри горутины.
- `wg.Done()` вызывается через `defer`.
- Нет `time.Sleep` для синхронизации.
- Воркер завершается именно потому что канал закрыт — а не по любой другой причине.
