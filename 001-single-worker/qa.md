## Блок 1: Канал

**Q: Что вернёт `v := <-ch` если канал закрыт и пуст?**

A: Вернёт zero value для типа канала. Для `chan string` — пустую строку `""`. Для `chan int` — `0`. Паники не будет.

---

**Q: Что вернёт `v, ok := <-ch` если канал закрыт и пуст?**

A: `v` — zero value, `ok` — `false`. Именно `ok == false` является надёжным признаком что канал закрыт. Без `ok` ты не можешь отличить закрытый канал от канала который просто передал zero value.

---

**Q: Почему `range` по каналу избавляет от проверки `ok` вручную?**

A: `for job := range ch` — это синтаксический сахар для:
```go
for {
    job, ok := <-ch
    if !ok {
        break
    }
    // использование job
}
```
`range` сам проверяет `ok` и завершает цикл когда канал закрыт и пуст.

---

**Q: Что произойдёт если не закрыть канал после отправки всех задач при использовании `range` в воркере?**

A: Воркер заблокируется навсегда на чтении из пустого незакрытого канала. `range` не завершится — он ждёт либо новых значений либо закрытия канала. Если никто не придёт и канал не закроется — горутина висит вечно. Если все горутины заблокированы — рантайм Go бросает `fatal error: all goroutines are asleep - deadlock!`.

---

**Q: Кто должен закрывать канал — отправитель или получатель?**

A: Всегда отправитель. Получатель никогда не закрывает канал. Причина: только отправитель знает что больше не будет новых данных. Кроме того, отправка в закрытый канал вызывает панику — если получатель закрывает канал пока отправитель ещё работает, получишь `panic: send on closed channel`.

---

**Q: Что такое рандеву в контексте небуферизованного канала?**

A: Небуферизованный канал требует чтобы отправитель и получатель встретились одновременно. Отправитель блокируется на `ch <- v` пока получатель не заберёт значение. Получатель блокируется на `<-ch` пока отправитель не отправит. Передача данных происходит в момент встречи — как из рук в руки.

---

**Q: Чем небуферизованный канал отличается от буферизованного с точки зрения поведения отправителя?**

A: При небуферизованном канале `ch <- v` блокируется до тех пор пока получатель не заберёт значение. При буферизованном `ch <- v` блокируется только если буфер полон — иначе значение кладётся в буфер и отправитель продолжает немедленно не ожидая получателя.

## Блок 2: Горутины

**Q: Что произойдёт с горутинами если `main` завершится?**

A: Все горутины будут убиты немедленно независимо от того на каком этапе они находятся. Процесс Go живёт пока жив `main`. Горутины не имеют независимого времени жизни — они не потоки ОС и не могут пережить завершение `main`.

---

**Q: Горутина захватывает переменную из внешней области видимости. По значению или по ссылке?**

A: По ссылке — она захватывает саму переменную (указатель на ячейку памяти), а не её значение в момент захвата. Если переменная изменится до того как горутина её прочитает — горутина увидит новое значение.

---

**Q: Что напечатает этот код и почему?**
```go
for i := 0; i < 3; i++ {
    go func() {
        fmt.Println(i)
    }()
}
time.Sleep(time.Second)
```

A: Скорее всего напечатает `3 3 3`. Все три горутины захватывают одну и ту же переменную `i`. К моменту когда горутины реально выполняются, цикл уже завершился и `i == 3`. Это классический баг захвата переменной цикла. Исправление: `i := i` в теле цикла или передача через параметр функции.

---

**Q: Как правильно захватить переменную цикла в горутине?**

A: Два способа. Первый — shadowing:
```go
for i := 0; i < 3; i++ {
    i := i // новая переменная на каждой итерации
    go func() { fmt.Println(i) }()
}
```
Второй — параметр функции:
```go
for i := 0; i < 3; i++ {
    go func(n int) { fmt.Println(n) }(i) // i копируется здесь
}
```

## Блок 3: WaitGroup

**Q: Почему `wg.Add(1)` должен вызываться до `go`, а не внутри горутины?**

A: Если `wg.Add(1)` внутри горутины — между запуском `go` и выполнением `wg.Add(1)` планировщик может переключиться на другую горутину. Если в этот момент выполнится `wg.Wait()` — счётчик равен нулю и `Wait` разблокируется преждевременно, не дождавшись реального завершения горутины.

---

**Q: Почему `wg.Done()` нужно вызывать через `defer`?**

A: `defer` гарантирует вызов при любом завершении функции — в том числе при панике. Без `defer`: если горутина паникует до достижения `wg.Done()` — счётчик никогда не уменьшится и `wg.Wait()` заблокируется навсегда.

---

**Q: `wg.Done()` — это сокращение для чего?**

A: Для `wg.Add(-1)`. Это явно уменьшает счётчик WaitGroup на единицу. Если вызвать `Done()` больше раз чем `Add()` — счётчик уйдёт в отрицательное значение и рантайм бросит панику: `sync: negative WaitGroup counter`.

---

**Q: Что произойдёт если передать WaitGroup в функцию по значению (не по указателю)?**

A: Функция получит копию WaitGroup. Вызов `wg.Done()` на копии не уменьшит счётчик оригинала в `main`. `wg.Wait()` в `main` будет ждать вечно. Поэтому WaitGroup всегда передаётся по указателю: `*sync.WaitGroup`.

---

**Q: Можно ли переиспользовать WaitGroup после того как `Wait` вернулся?**

A: Да, но только если нет конкурентных вызовов `Add` в момент когда `Wait` уже вернулся. Безопасный паттерн: `Wait` вернулся → все горутины завершены → можно вызывать `Add` снова для новой группы горутин.

## Блок 4: defer и close

**Q: В каком порядке выполняются несколько `defer` в одной функции?**

A: В обратном порядке объявления — LIFO (last in, first out). Последний объявленный `defer` выполняется первым.

---

**Q: `defer close(ch)` объявлен в горутине-отправителе. Когда именно вызовется `close`?**

A: В момент когда горутина-отправитель завершает выполнение своей функции — независимо от причины: нормальный return, паника, или выход из цикла. Это гарантирует что канал будет закрыт даже если в теле функции произойдёт что-то неожиданное.

---

**Q: Что произойдёт при попытке закрыть уже закрытый канал?**

A: Паника: `panic: close of closed channel`. Закрыть канал можно только один раз. Для защиты от двойного закрытия используют `sync.Once`.

---

**Q: Что произойдёт при отправке в закрытый канал?**

A: Паника: `panic: send on closed channel`. Это невосстановимая ошибка если не стоит `recover`. Именно поэтому закрывает канал всегда отправитель — только он знает что больше не будет отправлять.

## Блок 5: Финальные вопросы

**Q: Горутина и поток ОС — в чём принципиальная разница?**

A: Поток ОС создаётся операционной системой, имеет фиксированный стек (обычно 1-8MB), и переключение между потоками — системный вызов. Горутина создаётся рантаймом Go, начинает с 2KB стека (растёт динамически), и переключение управляется планировщиком Go в userspace без системных вызовов. Поэтому горутины в тысячи раз дешевле потоков.

---

**Q: Сколько горутин можно запустить одновременно в Go программе?**

A: Технически — миллионы. Практически ограничено доступной памятью (каждая горутина минимум 2KB стека) и числом задач которые имеет смысл выполнять параллельно. Нет жёсткого системного лимита как у потоков ОС.

---

**Q: Что такое `GOMAXPROCS` и какое значение по умолчанию?**

A: `GOMAXPROCS` — число потоков ОС которые могут одновременно выполнять Go код. По умолчанию равно числу логических ядер CPU. Это число P (processor) в планировщике GMP. Увеличение сверх числа ядер обычно не даёт прироста производительности для CPU-bound задач.

---

**Q: Небуферизованный канал создан но никто его не читает. Отправитель делает `ch <- v`. Что происходит?**

A: Отправитель блокируется навсегда. Если это единственная горутина — рантайм детектирует дедлок. Если есть другие горутины которые не заблокированы — программа продолжает работать, но эта горутина висит — это утечка горутины.
