# 003 — Results Channel

## Контекст

В предыдущих задачах воркеры только печатали результат — они ничего не возвращали вызывающему коду. В реальном продакшне воркер должен **передать результат обратно**: число, структуру, ошибку. Горутина не может вернуть значение через `return` как обычная функция — это фундаментальное ограничение модели горутин в Go. Единственный идиоматичный способ передать данные из горутины наружу — через канал.

В этой задаче ты впервые построишь полную двустороннюю схему: `jobs` идёт от `main` к воркерам, `results` идёт от воркеров обратно к `main`. Это и есть скелет любого worker pool.

Здесь же ты впервые столкнёшься с **дедлоком** — намеренно. Одно из заданий требует его воспроизвести и объяснить, потому что понять дедлок на своём коде важнее чем прочитать о нём.

---

## Задача

Напиши программу в которой:

1. Тип `Result` — структура с единственным полем `Value int`.

2. Функция `startWorker(id int, jobs <-chan int, results chan<- Result, wg *sync.WaitGroup)` — запускается как горутина. Читает числа из `jobs`, возводит каждое в квадрат, отправляет `Result{Value: n * n}` в канал `results`. Когда `jobs` закрыт — завершается.

3. В `main`:
    - Создай канал `jobs` буферизованный на 9 элементов.
    - Создай канал `results` буферизованный на количество воркеров (3).
    - Запусти 3 воркера.
    - Отправь числа от 1 до 9 в `jobs`.
    - Закрой `jobs`.
    - Запусти **горутину-надсмотрщика** которая ждёт завершения всех воркеров через WaitGroup и закрывает `results`.
    - В основной горутине собери все результаты из `results` через `range`, посчитай их сумму и напечатай.

---

## Требования

- Воркер принимает `results chan<- Result` — однонаправленный канал только для записи. Это явная документация намерений: воркер пишет, не читает.
- `results` закрывает **горутина-надсмотрщик** — не `main` напрямую и не сам воркер.
- Получатель читает `results` через `range` — он должен корректно завершиться когда все результаты собраны.
- Сумма квадратов чисел от 1 до 9 равна 285 — это твой контрольный ответ.
- Никаких `time.Sleep` для синхронизации.

## Ожидаемый вывод

```
result: 1
result: 9
result: 4
result: 25
result: 16
...
sum: 285
main done
```

Порядок результатов не определён — воркеры работают конкурентно. Сумма всегда 285.

## Вопросы ДО написания кода

**Вопрос 1.**  
Горутина не может вернуть значение через `return`. Почему это так устроено — в чём принципиальное отличие горутины от обычной функции с точки зрения времени жизни и стека вызовов?

**Вопрос 2.**  
Кто должен закрывать канал `results` — и почему именно он, а не кто-то другой?  
Рассмотри три варианта и объясни что пойдёт не так в каждом неправильном:

```go
// Вариант А: main закрывает results сразу после close(jobs)
close(jobs)
close(results) // сразу
for r := range results { ... }

// Вариант Б: каждый воркер закрывает results когда завершается
defer close(results) // внутри воркера

// Вариант В: отдельная горутина-надсмотрщик закрывает results после wg.Wait()
go func() {
    wg.Wait()
    close(results)
}()
```

**Вопрос 3.**  
Почему буфер канала `results` стоит делать равным числу воркеров, а не числу задач?  
Что происходит если буфер `results` равен нулю (небуферизованный)?  
При каком условии воркер заблокируется на отправке в `results`?

**Вопрос 4.**  
Рассмотри этот код в `main` — найди дедлок и объясни почему он возникает:

```go
// Отправляем все задачи
for i := 1; i <= 9; i++ {
    jobs <- i
}
close(jobs)

// Ждём воркеров
wg.Wait()

// Читаем результаты
for r := range results {
    fmt.Println(r.Value)
}
```

Нарисуй в голове что происходит пошагово: воркеры обрабатывают задачи, пишут в `results`. Буфер `results` равен 3. После 3 результатов что происходит с воркерами? Кто их разблокирует?

**Вопрос 5.**  
`chan<- Result` и `<-chan Result` — в чём разница и зачем Go позволяет ограничивать направление канала?  
Что произойдёт если внутри функции принимающей `chan<- Result` попробовать прочитать из этого канала?

**Вопрос 6.**  
Горутина-надсмотрщик делает `wg.Wait()` а потом `close(results)`. Эта горутина сама никогда не попадает в WaitGroup — её никто не ждёт явно. Как тогда `main` узнает что `results` закрыт и пора выходить из `range`? Проследи цепочку событий от `close(jobs)` до выхода из `range results` в `main`.

## Эксперимент: намеренный дедлок

Это обязательная часть задачи, не опциональная.

Напиши **намеренно неправильную** версию `main` где ты:
1. Отправляешь все задачи в `jobs`.
2. Закрываешь `jobs`.
3. Вызываешь `wg.Wait()` — ждёшь завершения воркеров.
4. Только потом читаешь из `results`.

Запусти эту версию. Убедись что программа зависает. Посмотри на вывод `go run` — Go детектирует дедлок и печатает `fatal error: all goroutines are asleep - deadlock!`.

Теперь ответь письменно: на каком именно шаге возникает дедлок? Какая горутина заблокирована и на чём? Почему `wg.Wait()` никогда не разблокируется в этой версии?

Только после того как ты воспроизвёл и объяснил дедлок — исправь код до правильной версии.

## Вопросы ПОСЛЕ решения

**Вопрос А.**  
Ты использовал однонаправленные каналы `chan<- Result` и `<-chan Result`. Go позволяет неявно конвертировать двунаправленный `chan Result` в однонаправленный при передаче в функцию. Работает ли обратное преобразование — из однонаправленного в двунаправленный? Почему?

**Вопрос Б.**  
Надсмотрщик — это горутина которая ничего не делает кроме ожидания и закрытия канала. Это кажется избыточным. Можно ли обойтись без неё? Придумай альтернативный дизайн где `results` закрывается без отдельной горутины — и объясни его компромиссы.

**Вопрос В.**  
Сумма квадратов от 1 до 9 равна 285. Ты проверяешь это в конце. Но что если один воркер молча потеряет результат — например из-за бага — и сумма окажется меньше? Какие инструменты Go позволяют детектировать такие ситуации на этапе тестирования?

## Критерии правильного решения

Твоё решение корректно если: программа завершается без дедлока и без зависания; сумма результатов равна 285; `results` закрывается горутиной-надсмотрщиком после `wg.Wait()` — не раньше и не позже; воркер принимает `results` как `chan<- Result` — только запись; `range results` в `main` завершается естественно через закрытие канала — без дополнительных флагов или счётчиков.
