# 007 — Graceful Shutdown vs Stop: Два способа завершения

## Контекст

К этому моменту твой воркер умеет многое: читать задачи, возвращать результаты, обрабатывать ошибки, выживать после паники, реагировать на контекст. Но у завершения работы пула есть два принципиально разных сценария которые требуют разного поведения — и смешивать их нельзя.

Первый сценарий — **Graceful Shutdown**: штатное завершение. Новых задач больше не будет, но те что уже лежат в очереди должны быть обработаны до последней. Именно так сервис должен себя вести при деплое: получил SIGTERM, перестал принимать новые запросы, дообработал текущие, завершился чисто. Потеря данных недопустима.

Второй сценарий — **Stop**: экстренное завершение. Немедленная остановка — неважно сколько задач осталось в очереди. Используется при критической ошибке, превышении лимита ресурсов, или когда оператор хочет остановить сервис прямо сейчас. Скорость важнее полноты обработки.

В этой задаче ты реализуешь оба метода в одной структуре `Pool`, поймёшь почему они используют разные механизмы (`close(jobs)` vs `cancel()`), и убедишься что дважды вызвать любой из них не приводит к панике.

---

## Задача

Напиши программу в которой:

1. Структура `Pool` с полями: `jobs chan Job`, `results chan Result`, `wg sync.WaitGroup`, `ctx context.Context`, `cancel context.CancelFunc`, `once sync.Once`.

2. Конструктор `NewPool(ctx context.Context, workers int, jobsBuf int) *Pool` — создаёт дочерний контекст, инициализирует каналы, запускает `workers` воркеров и горутину-надсмотрщика.

3. Метод `Submit(job Job) error` — отправляет задачу в `jobs`. Если пул остановлен или контекст отменён — возвращает ошибку. Не блокируется бесконечно: использует `select` с `ctx.Done()`.

4. Метод `Shutdown()` — graceful завершение. Закрывает канал `jobs` чтобы воркеры дочитали оставшиеся задачи и завершились. Ждёт завершения всех воркеров. Безопасен при повторном вызове — не паникует.

5. Метод `Stop()` — немедленная остановка. Отменяет контекст. Воркеры выходят через `ctx.Done()` не дожидаясь обработки задач в очереди. Ждёт завершения всех воркеров. Безопасен при повторном вызове.

6. Метод `Results() <-chan Result` — возвращает read-only канал результатов.

7. В `main` — реализуй два сценария последовательно и покажи разницу в поведении.

---

## Требования

- `Shutdown` использует `close(jobs)` — не `cancel()`. Это принципиально: закрытие канала позволяет воркерам дочитать очередь, отмена контекста — нет.
- `Stop` использует `cancel()` — не `close(jobs)`. После отмены контекста воркеры выходят через `ctx.Done()` немедленно.
- Повторный вызов `Shutdown()` не должен вызывать `panic: close of closed channel`. Используй `sync.Once`.
- Повторный вызов `Stop()` не должен вызывать панику. `cancel()` идемпотентен — повторный вызов безопасен.
- `Shutdown` и `Stop` можно вызывать в любом порядке — программа не должна падать.
- После `Shutdown` или `Stop` вызов `Submit` должен возвращать ошибку, а не зависать.

## Сценарии в main

**Сценарий А — Graceful Shutdown:**

Создай пул с 3 воркерами. Отправь 12 задач. Каждая задача занимает `30ms` (имитация работы). Вызови `Shutdown()`. Убедись что **все 12 задач обработаны**. Напечатай количество обработанных задач.

```
[Scenario A: Graceful Shutdown]
submitted 12 jobs
calling Shutdown...
all workers finished processing the queue
processed: 12 / 12
```

**Сценарий Б — Stop:**

Создай новый пул с 3 воркерами. Отправь 12 задач с той же задержкой 30ms. Подожди `50ms` (пусть несколько задач успеют обработаться). Вызови `Stop()`. Убедись что программа завершилась **не дождавшись всех 12 задач**. Напечатай сколько успело обработаться.

```
[Scenario B: Stop]
submitted 12 jobs
waiting 50ms then calling Stop...
workers stopped immediately
processed: 4 / 12
```

Точное число в Б зависит от планировщика — важно что оно **меньше 12**.

## Вопросы ДО написания кода

**Вопрос 1.**
Почему `Shutdown` и `Stop` используют разные механизмы? Что конкретно произойдёт с задачами в буфере `jobs` если вместо `close(jobs)` в `Shutdown` вызвать `cancel()`? А если вместо `cancel()` в `Stop` вызвать `close(jobs)` — воркеры остановятся немедленно?

**Вопрос 2.**
`close(jobs)` можно вызвать только один раз — повторный вызов паникует. При этом `Shutdown` может быть вызван несколько раз — например из разных горутин при получении сигнала. Как `sync.Once` решает эту проблему? Покажи как выглядит `Once.Do` под капотом — что он гарантирует при конкурентных вызовах?

**Вопрос 3.**
После вызова `Shutdown()` — закрыт ли канал `jobs` немедленно? Или `Shutdown` ждёт пока воркеры его дочитают? Нарисуй последовательность событий: `close(jobs)` → воркеры читают оставшиеся задачи → воркеры выходят из `range` → `wg.Done()` → `wg.Wait()` разблокируется → надсмотрщик закрывает `results` → `range results` в `main` завершается.

**Вопрос 4.**
В `Submit` ты используешь `select` с `ctx.Done()`. Но к моменту когда `Submit` вызывается, контекст мог уже быть отменён. Как `select` ведёт себя если `ctx.Done()` уже закрыт в момент входа в `select`? Гарантированно ли он выберет `ctx.Done()` или может случайно выбрать `jobs`?

**Вопрос 5.**
Рассмотри этот код и найди гонку данных:

```go
type Pool struct {
    stopped bool
    jobs    chan Job
}

func (p *Pool) Submit(job Job) error {
    if p.stopped { // читаем без блокировки
        return errors.New("pool is stopped")
    }
    p.jobs <- job
    return nil
}

func (p *Pool) Shutdown() {
    p.stopped = true // пишем без блокировки
    close(p.jobs)
}
```

Что произойдёт если `Submit` и `Shutdown` вызываются конкурентно? Почему флаг `stopped` без защиты — это гонка данных? Как твоя реализация через `select` и `ctx.Done()` избегает этой проблемы?

**Вопрос 6.**
`wg.Wait()` в `Shutdown` блокирует вызывающую горутину пока все воркеры не завершатся. Если воркеры обрабатывают задачи по 30ms каждая и в очереди 100 задач — как долго может блокироваться `Shutdown`? В реальном сервисе это проблема? Как её решают?

## Эксперимент: опасность двойного close

Это обязательная часть задачи — выполни до написания финального решения.

Напиши простую программу без `sync.Once`:

```go
jobs := make(chan int, 5)
close(jobs)
close(jobs) // что происходит?
```

Запусти. Зафиксируй вывод. Теперь добавь `sync.Once`:

```go
var once sync.Once
jobs := make(chan int, 5)
closeJobs := func() { once.Do(func() { close(jobs) }) }
closeJobs()
closeJobs() // безопасно?
```

Запусти. Зафиксируй что изменилось. Потом вызови `closeJobs` из двух горутин одновременно в цикле 1000 раз — убедись что паники нет ни разу. Это и есть гарантия `sync.Once`.

## Вопросы ПОСЛЕ решения

**Вопрос А.**
В реальном HTTP сервере на Go graceful shutdown обычно выглядит так: получили `SIGTERM`, вызвали `server.Shutdown(ctx)`, ждём завершения текущих запросов. Как твоя реализация `Pool.Shutdown()` соотносится с этим паттерном? Что общего в механизме?

**Вопрос Б.**

В продакшне `Shutdown` часто вызывается с таймаутом: "завершись gracefully, но не дольше 30 секунд". Как модифицировать `Shutdown` чтобы он принимал `context.Context` и завершался по таймауту даже если воркеры ещё не закончили?

## Критерии правильного решения

Твоё решение корректно если выполняются все условия одновременно: в сценарии А все 12 задач обработаны до завершения; в сценарии Б программа завершается не дожидаясь всех задач; повторный вызов `Shutdown()` не паникует — защищён `sync.Once`; повторный вызов `Stop()` не паникует — `cancel()` идемпотентен; `Submit` после остановки возвращает ошибку а не блокируется; детектор гонок `go run -race .` молчит на обоих сценариях.
