# 002 — Multiple Workers

## Контекст

В предыдущей задаче один воркер обрабатывал задачи последовательно.  
Теперь мы запускаем нескольких воркеров которые **конкурируют** за задачи из одного канала.  
Это фундаментальная идея worker pool: один канал — много читателей, каждую задачу получает ровно один.

Здесь ты впервые увидишь недетерминированность в действии: порядок вывода будет разным при каждом запуске. Это не баг — это правильное поведение конкурентной программы.

---

## Задача

Напиши программу в которой:

1. Функция `startWorker(id int, jobs <-chan string, wg *sync.WaitGroup)` — та же что в задаче 001, но теперь воркеров будет несколько.

2. В `main`:
    - Создай буферизованный канал `jobs` с размером буфера 9.
    - Запусти **3 воркера** с id 0, 1, 2.
    - Отправь в канал **9 задач**: `"task-0"` ... `"task-8"`.
    - Закрой канал.
    - Дождись завершения всех трёх воркеров.
    - Напечатай `"main done"`.

---

## Требования

- Один `sync.WaitGroup` на всех воркеров — не создавай отдельный WaitGroup для каждого.
- `wg.Add(1)` вызывается перед каждым `go` — в цикле, не внутри горутины.
- Все 9 задач должны быть обработаны: каждую задачу получает ровно один воркер.
- Программа завершается без зависания.
- Никаких `time.Sleep` для синхронизации.

---

## Ожидаемый вывод

Порядок строк между воркерами **не определён** и меняется при каждом запуске.  
Но каждая из 9 задач должна появиться ровно один раз. Пример одного из возможных выводов:

```
worker-0 processing: task-0
worker-1 processing: task-1
worker-0 processing: task-2
worker-2 processing: task-3
worker-1 processing: task-4
worker-2 processing: task-5
worker-0 processing: task-6
worker-1 processing: task-7
worker-2 processing: task-8
worker-0 done
worker-2 done
worker-1 done
main done
```

## Вопросы ДО написания кода

**Вопрос 1.**  
Три воркера читают из одного канала одновременно. Как Go гарантирует что одну задачу получит ровно один воркер — а не все три сразу? Что происходит под капотом когда несколько горутин стоят на чтении из одного канала?

**Вопрос 2.**  
В задаче 001 канал был небуферизованным. Здесь он буферизованный с cap=9.  
Почему в этой задаче буферизованный канал удобнее?  
Что произошло бы если оставить небуферизованный канал и попробовать отправить все 9 задач до запуска воркеров?

**Вопрос 3.**  
Сколько раз нужно вызвать `wg.Add` и `wg.Done` в этой задаче?  
Нарисуй в голове счётчик WaitGroup на каждом шаге программы: после трёх `wg.Add(1)`, после завершения первого воркера, второго, третьего. Что происходит когда счётчик достигает нуля?

**Вопрос 4.**  
Посмотри на этот код и найди баг:

```go
for i := 0; i < 3; i++ {
    go startWorker(i, jobs, &wg)
    wg.Add(1)
}
```

При каких условиях этот баг проявится? Он проявится всегда или только иногда?

**Вопрос 5.**  
Посмотри на этот код и объясни что напечатают воркеры:

```go
for i := 0; i < 3; i++ {
    go func() {
        fmt.Println("worker id:", i)
    }()
}
time.Sleep(time.Second)
```

Почему это происходит? Как это связано с тем как замыкания захватывают переменные?  
Как ты защитишься от этого в своём решении?

**Вопрос 6.**  
После того как `main` вызовет `close(jobs)` — что произойдёт с воркерами которые в этот момент заблокированы на чтении из пустого канала? Они получат панику? Zero value? Или что-то ещё?

Стал ли порядок получения задач более предсказуемым? Почему? Является ли этот порядок гарантированным?

## Критерии правильного решения

Твоё решение корректно если выполняются все условия одновременно: все 9 задач обработаны ровно по одному разу, ни одна задача не потеряна и не обработана дважды; программа завершается чисто без дедлока и без зависания; `wg.Add(1)` вызывается до `go` в каждой итерации цикла; воркеры получают `id` корректно — без классического бага захвата переменной цикла; `wg.Done()` вызывается через `defer` внутри каждого воркера.
